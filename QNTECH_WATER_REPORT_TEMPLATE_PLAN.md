# QnTECH 스타일 수질분석일지 양식 전환 계획

## 목적

최초 배포 버전 안정화가 끝난 뒤, 다음 버전 업그레이드에서 `수질분석일지` 출력 양식을 QnTECH 일일 보고서 스타일에 가깝게 전환한다.

이번 계획은 즉시 구현 대상이 아니다. 최초 배포 파일을 완전히 점검하고 배포한 뒤, 다음 개발 사이클에서 적용한다.

## 현재 확인된 사실

- 앱은 이미 QnTECH에서 수질 분석값과 사진을 가져오는 기능을 갖고 있다.
- 현재 QnTECH 연동은 보고서 양식 자체를 가져오는 구조가 아니라, GraphQL로 분석 데이터와 파일 경로를 가져온 뒤 앱 내부 Excel 템플릿에 바인딩하는 구조다.
- 사용자가 직접 수정한 QnTECH 스타일 후보 템플릿 파일이 있다.
  - `C:\Users\ASUS\OneDrive\바탕 화면\수질분석일지.xlsx`
- 해당 후보 템플릿은 기존 기본 템플릿보다 실제 보고서 레이아웃에 가까우며, 시트 안에 이름범위와 이미지/그래프 영역이 포함되어 있다.

## 관련 코드 위치

- QnTECH 값 가져오기
  - `server/services/qntechWaterValueImportService.cjs`
- QnTECH 사진 가져오기
  - `server/services/qntechWaterPhotoImportService.cjs`
- 수질분석일지 Excel/PDF 출력 바인딩
  - `server/services/dailyLogPreviewService.cjs`
- 수질분석일지 출력 API
  - `server/routes/excelRoutes.cjs`
- 보고서 템플릿 검색/업로드
  - `server/services/reportTemplateService.cjs`
  - `server/routes/settingsRoutes.cjs`
- 기본 번들 템플릿 위치
  - `templates/reports/수질분석일지.xlsx`
- 실행 중 사용자 템플릿 위치
  - `%APPDATA%\Osoo_Handle_App\templates\reports\수질분석일지.xlsx`

## 후보 템플릿 점검 결과

후보 파일은 대략 다음 이름범위를 가진다.

- `날짜`
- `암모니아`
- `질산성`
- `인산염`
- `알칼리`
- `암모니아사진`
- `질산사진`
- `인사진`
- `알칼리도사진`
- `암모니아그래프`
- `질산그래프`
- `인산염그래프`
- `알칼리도그래프`
- `인_알칼리그래프`
- `암모니아_질산그래프`

주의할 점:

- 기존 바인딩 코드는 `암모니아1`, `암모니아2`처럼 단일 셀 이름범위에 값을 넣는 방식에 가깝다.
- 후보 템플릿은 `암모니아`, `질산성`, `인산염`, `알칼리`처럼 항목별 행 범위에 여러 현장 위치 값을 넣는 방식이다.
- 따라서 템플릿만 교체하면 값 바인딩이 완전하지 않을 수 있다.
- 사진 위치도 기존 고정 anchor 보정 로직이 있어서 새 양식의 사진 병합 셀 위치와 충돌할 수 있다.

## 권장 구현 방향

### 1. 템플릿 파일 정리

1. 사용자가 만든 후보 파일을 다음 버전용 원본으로 보관한다.
2. 후보 파일의 파일명은 최종적으로 `수질분석일지.xlsx`를 유지한다.
3. 최초 배포 전에는 현재 기본 템플릿을 변경하지 않는다.
4. 다음 버전 작업 때만 `templates/reports/수질분석일지.xlsx`를 교체한다.

### 2. 측정장소 기반 데이터 바인딩 코드 보강

QnTECH 스타일 양식에서는 측정항목 행과 측정장소 열이 교차하는 표 구조를 사용한다.

따라서 템플릿 작성자가 `암모니아1`, `암모니아2`처럼 측정장소별 이름범위를 모두 수작업으로 넣지 않아도, 코드가 다음 기준으로 값을 매핑하도록 한다.

#### 기본 매핑 원칙

1. 측정항목은 행으로 판단한다.
   - 암모니아성질소 → `nh3_n`
   - 질산성질소 → `no3_n`
   - 오르토인산염 → `po4_p`
   - 알칼리도 → `alkalinity`
2. 측정장소는 열 순서로 판단한다.
3. 첫 번째 측정장소는 항상 `유량조정조`로 본다.
4. 마지막 측정장소는 항상 `방류조`로 본다.
5. 중간 측정장소는 현장 공법과 설정값에 따라 달라질 수 있으므로, 설정 메뉴에서 관리하는 측정장소 목록을 사용한다.

#### 기본 열 순서 예시

현재 후보 템플릿의 기본 표 구조는 다음 순서를 기준으로 한다.

| 위치 번호 | 기본 측정장소 | 비고 |
| --- | --- | --- |
| 1 | 유량조정조 | 고정 |
| 2 | 혐기조 | 설정 가능 |
| 3 | 무산소조 | 설정 가능 |
| 4 | 포기조 | 설정 가능 |
| 5 | 침전조 | 설정 가능 |
| 6 | 방류조 | 고정 |

현장에 따라 중간 공정이 달라지면 `config_items`의 `location` 설정 또는 향후 설정 메뉴의 수질분석 장소 설정값을 기준으로 이 순서를 재구성한다.

#### 코드 구현 계획

`dailyLogPreviewService.cjs`에 다음 로직을 추가한다.

1. 후보 템플릿의 표 헤더 행에서 측정장소 병합 셀을 읽는다.
2. 측정장소명이 비어 있거나 모두 같은 이름으로 잘못 지정되어 있어도, 열 위치 기준으로 장소 번호를 계산한다.
3. `getActiveLocations(db)` 결과를 기준으로 장소 번호와 실제 DB `location` 값을 매칭한다.
4. 후보 템플릿처럼 `암모니아`, `질산성`, `인산염`, `알칼리` 이름범위가 행 전체를 가리키면, 해당 행의 병합 셀 구간을 측정장소 1~N으로 해석한다.
5. 각 구간에 다음 값을 채운다.
   - `rowsByLocation['유량조정조'].nh3_n`
   - `rowsByLocation['혐기조'].nh3_n`
   - `rowsByLocation['무산소조'].nh3_n`
   - 같은 방식으로 항목별 필드 적용
6. 기존 `암모니아1`, `질산1` 같은 단일 셀 이름범위 방식도 그대로 유지한다.

#### 의사 코드

```javascript
const activeLocations = getActiveLocations(db);
const locationSlots = normalizeWaterReportLocationSlots(activeLocations);
const rowsByLocation = new Map(rows.map((row) => [row.location, row]));

for (const namedRange of namedRanges) {
  if (namedRange.name === '암모니아') {
    bindAnalyteRow(namedRange, 'nh3_n', locationSlots, rowsByLocation);
  }
}
```

`normalizeWaterReportLocationSlots()`는 다음 규칙을 따른다.

```javascript
function normalizeWaterReportLocationSlots(activeLocations) {
  const middle = activeLocations
    .filter((location) => location !== '유량조정조' && location !== '방류조');

  return [
    '유량조정조',
    ...middle,
    '방류조'
  ].slice(0, 6);
}
```

추후 현장별로 측정장소가 6개보다 많거나 적은 경우까지 고려하려면, 템플릿의 병합 셀 개수와 설정 장소 개수를 비교하여 경고를 띄우는 검증 로직도 추가한다.

### 3. 기존 이름범위 방식과의 호환성 유지

`dailyLogPreviewService.cjs`에 다음 기능을 추가한다.

1. Excel 이름범위가 단일 셀뿐 아니라 `E4:AB4` 같은 범위도 인식하도록 확장한다.
2. `암모니아`, `질산성`, `인산염`, `알칼리` 이름범위를 항목별 행 바인딩으로 해석한다.
3. 해당 행 안의 병합 셀 구간을 현장 위치 순서로 매칭한다.
4. `getActiveLocations()`에서 가져온 현장 위치 순서를 기준으로 값을 채운다.
5. 기존 `암모니아1`, `질산1` 같은 단일 셀 방식도 유지하여 과거 템플릿과 호환성을 지킨다.

### 4. 사진 바인딩 보강

1. 새 양식의 사진 이름범위인 `암모니아사진`, `질산사진`, `인사진`, `알칼리도사진`을 계속 사용한다.
2. 기존 고정 사진 anchor 로직은 구버전 템플릿 전용으로 제한한다.
3. 새 템플릿에서는 이름범위가 가리키는 병합 셀 크기에 맞춰 사진을 `cover` 방식으로 삽입한다.
4. 사진이 없는 경우 셀을 비워두고 출력 실패로 처리하지 않는다.

### 5. 그래프 처리 전략

1차 구현에서는 Excel 내장 차트/이미지 레이아웃을 최대한 유지한다.

권장 방식:

- 표 데이터 영역을 채우면 Excel 차트가 참조하는 범위가 자동 갱신되도록 템플릿을 구성한다.
- 앱 코드에서 차트 이미지를 직접 생성하지 않는다.
- PDF 변환 전 Excel이 차트를 정상 렌더링하는지 확인한다.
- 사용자가 더미 데이터를 직접 제거한 빈 양식 파일을 최종 템플릿으로 제공한다.
- 다음 버전 업그레이드 작업 때 이 완성된 빈 양식으로 기존 `수질분석일지.xlsx`를 교체한다.
- 그래프는 빈 데이터 범위를 참조한 초기 상태로 유지하고, 출력 시 앱이 데이터 셀을 채우면 자동으로 다시 그려지게 한다.

추가 검토:

- ExcelJS가 차트 객체를 완전히 보존하지 못하는 경우가 있으면, 차트를 이미지로 고정하거나 숨김 데이터 영역 + 기존 차트 보존 방식을 재검토한다.

### 6. 최종 구현 범위: 데이터 셀 위치 매핑 + 사진 삽입

다음 버전의 실제 구현 범위는 의도적으로 작게 잡는다.

앱이 담당하는 것:

1. 수질 데이터 표의 값 셀 채우기
2. QnTECH 측정 사진 4종 삽입
3. 날짜/차수 같은 최소 메타데이터 채우기
4. 템플릿 구조 검증 및 오류 메시지 제공

앱이 담당하지 않는 것:

1. 그래프 생성
2. 그래프 스타일 수정
3. 그래프 위치 조정
4. 차트 축/범례/색상 제어
5. QnTECH 보고서 화면을 직접 캡처하거나 변환

#### 데이터 셀 위치 매핑 상세

템플릿의 데이터 표는 다음처럼 해석한다.

| 측정항목 행 | DB 필드 | 바인딩 방식 |
| --- | --- | --- |
| 암모니아성질소 | `nh3_n` | 측정장소 열 순서대로 값 입력 |
| 질산성질소 | `no3_n` | 측정장소 열 순서대로 값 입력 |
| 오르토인산염 | `po4_p` | 측정장소 열 순서대로 값 입력 |
| 알칼리도 | `alkalinity` | 측정장소 열 순서대로 값 입력 |

측정장소 열 순서는 다음 규칙으로 만든다.

1. 설정 메뉴의 활성 측정장소 목록을 읽는다.
2. `유량조정조`는 항상 첫 번째 위치로 강제한다.
3. `방류조`는 항상 마지막 위치로 강제한다.
4. 나머지 중간 장소는 설정 메뉴의 표시 순서를 따른다.
5. 템플릿의 데이터 셀 슬롯 개수와 설정 장소 개수가 다르면 출력 전 경고한다.

#### 후보 템플릿 이름범위 권장안

템플릿에는 다음 이름범위를 권장한다.

| 이름범위 | 용도 |
| --- | --- |
| `날짜` | 출력 날짜 |
| `암모니아` | 암모니아성질소 값이 들어갈 행 범위 |
| `질산성` | 질산성질소 값이 들어갈 행 범위 |
| `인산염` | 오르토인산염 값이 들어갈 행 범위 |
| `알칼리` | 알칼리도 값이 들어갈 행 범위 |
| `암모니아사진` | 암모니아성질소 측정 사진 |
| `질산사진` | 질산성질소 측정 사진 |
| `인사진` | 오르토인산염 측정 사진 |
| `알칼리도사진` | 알칼리도 측정 사진 |

값 이름범위는 단일 셀이 아니라 행 범위여도 된다. 예를 들어 `암모니아 = E4:AB4`처럼 지정하면, 코드가 해당 행 안의 병합 셀 구간을 측정장소 슬롯으로 해석한다.

#### 사진 삽입 상세

사진 삽입은 다음 규칙으로 처리한다.

1. QnTECH에서 내려받은 사진 파일명/항목명을 기준으로 4개 항목을 매칭한다.
2. 각 항목의 이름범위가 가리키는 병합 셀 또는 단일 셀 영역을 찾는다.
3. 사진은 해당 영역을 꽉 채우되 비율을 유지하는 `cover` 방식으로 삽입한다.
4. 사진이 없으면 빈 칸으로 둔다.
5. 사진 누락은 출력 실패가 아니라 경고로 처리한다.

#### 구현 검증 기준

다음 조건을 만족하면 구현 완료로 본다.

1. 더미 데이터를 모두 지운 템플릿에서 출력해도 표 값이 정확히 채워진다.
2. 그래프가 앱 코드 수정 없이 표 값을 반영해 자동 갱신된다.
3. 사진 4종이 각 사진 영역에 정확히 들어간다.
4. 현장별 측정장소 설정이 바뀌어도 첫 칸은 `유량조정조`, 마지막 칸은 `방류조`로 유지된다.
5. 중간 공정명이 달라져도 설정 메뉴의 측정장소 순서를 따라 데이터가 들어간다.
6. 기간 출력으로 여러 시트가 생성되어도 각 시트의 표/사진/그래프가 독립적으로 정상 출력된다.

### 7. QnTECH 원본 보고서 직접 가져오기 검토

후순위로 다음 가능성을 확인한다.

1. QnTECH 웹사이트에 일일 보고서 HTML/PDF 다운로드 API가 있는지 확인한다.
2. 현재 `selectProjectListByRegDt` 외에 보고서 조회용 GraphQL 쿼리 또는 REST URL이 있는지 개발자 도구로 확인한다.
3. 실제 보고서 파일을 받을 수 있다면, 앱 내부 템플릿 생성 방식과 비교한다.

다만 안정성 측면에서는 앱 내부 템플릿 방식이 더 안전하다. QnTECH 웹 화면 구조가 바뀌어도 앱 출력이 흔들리지 않기 때문이다.

## 배포 전략

1. 다음 버전 개발 브랜치에서만 작업한다.
2. 최초 배포 버전에는 포함하지 않는다.
3. 자동 업데이트 배포 전, 개발 PC에서 AppData 템플릿까지 갱신되는지 확인한다.
4. 기존 사용자 템플릿이 AppData에 남아 있으면 번들 템플릿이 자동 덮어쓰기 되지 않을 수 있으므로, 템플릿 버전 관리 또는 강제 갱신 정책을 함께 검토한다.

## 리스크

- ExcelJS가 차트 객체를 보존하지 못하면 그래프 출력이 깨질 수 있다.
- 새 템플릿이 행 범위 이름범위를 사용하므로 기존 바인딩 방식과 다르다.
- AppData에 이미 사용자 템플릿이 있으면 새 번들 템플릿이 적용되지 않을 수 있다.
- 사진 anchor 보정 로직이 기존 템플릿에 맞춰져 있어 새 템플릿과 충돌할 수 있다.

## 결론

다음 버전에서는 QnTECH 원본 보고서를 직접 가져오는 방식보다, 사용자가 만든 QnTECH 스타일 Excel 템플릿을 앱의 기존 수질분석일지 출력 파이프라인에 맞춰 바인딩하는 방식이 가장 안정적이다.

핵심 작업은 템플릿 교체가 아니라, 새 템플릿의 범위형 이름범위와 사진 영역을 현재 출력 서비스가 정확히 이해하도록 바인딩 계층을 확장하는 것이다.
