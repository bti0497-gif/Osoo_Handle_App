# 리팩토링 계획

## 1. 목표

이번 리팩토링의 핵심 목표는 **앱 레이아웃과 워크스페이스를 분리**하는 것이다.

현재 버전에서는 왼쪽 메뉴 접기/펼치기 기능을 구현하지 않는다. 현장관리자가 먼저 기존 메뉴 구조에 익숙해질 수 있도록 현재 사용 흐름과 우측 워크스페이스의 기본 배치는 유지한다. 다만 다음 업그레이드에서 메뉴를 접고 아이콘만 표시하거나 숨겼을 때 워크스페이스를 화면 중앙에 배치할 수 있도록 내부 구조를 먼저 만든다.

리팩토링 완료 기준은 파일 구조 변경 자체가 아니라, 현장 설치 후 하루 업무가 끝날 때까지의 운영 워크플로우가 문제없이 동작하는 것이다.

```text
App
└─ AppShell
   ├─ Header
   ├─ NavigationArea
   │  ├─ AccountWidget
   │  └─ MenuTreeWidget
   ├─ WorkspaceArea
   │  └─ WorkspaceShell
   └─ StatusBar
```

## 2. 현재 전제

- 아직 실제 현장에 설치된 버전은 없다.
- GitHub에는 최초 릴리즈가 올라가 있지만, 실제 배포 전이므로 기존 최초 릴리즈 asset은 교체할 수 있다.
- 하위 호환성보다 설치 전 기준의 올바른 구조, 안정성, 유지보수성이 우선이다.
- 이미 공식 구조로 잡힌 `MVVM + feature` 구조는 유지한다.
- 구조 변경이 필요한 경우에도 기능 단위와 책임 분리는 명확히 한다.
- 이번 리팩토링은 현재 앱을 먼저 안정화하되, 이후 관리자 전용 앱을 별도 프로젝트로 분리할 수 있도록 관리자 기능의 경계를 명확히 한다.

## 3. 권한과 앱 영역

초기 권한 모델은 단순하게 유지한다.

- 현장관리자: 현장 운영 업무만 수행한다.
- 관리자 계열: 설정, 초기 데이터 입력, 현장/회원 관리, 서버/동기화 설정을 수행한다.
- 최고관리자와 중앙관리자 권한 세분화는 이후 확장 가능하도록 구조만 열어둔다.

현재 앱은 현장 설치 전 초기 설정과 현장 운영을 모두 포함하지만, 장기적으로는 **현장관리자용 앱**과 **관리자 전용 앱**을 분리한다.

현재 앱에서의 영역은 크게 두 가지로 나눈다.

```text
Admin Setup Area
├─ 현장 기본 설정
├─ 기존 엑셀 데이터 매핑/입력
├─ 회원 및 현장 관리
├─ 템플릿/일지 설정
└─ 외부 서비스/동기화 설정

Site Operation Area
├─ 오늘 업무 입력
├─ 현장 데이터 조회
├─ 일지 출력
├─ 접속자 정보
└─ 동기화 상태
```

현장관리자는 설정 관련 메뉴를 볼 수 없어야 한다.

장기 분리 전략:

- 현재 버전에서는 기존처럼 관리자 설정 메뉴를 포함해 구현한다.
- 단, 관리자 기능은 나중에 별도 프로젝트로 복사/이관하기 쉽도록 `features/admin`, `features/members`, `features/certificate` 같은 책임 경계를 명확히 유지한다.
- 현장관리자용 앱에 최종적으로 남길 기능은 현장 업무 입력, 조회, 일지 출력, 시설관리, 현장 범위 소통게시판, 성적서 조회/다운로드이다.
- 관리자 전용 앱으로 옮길 기능은 회원관리, 전체 게시판 관리, BigQuery 테이블 조회/수정/삭제, 관리자 대시보드, 성적서 ZIP 업로드/관리이다.
- 분리 전까지는 한 코드베이스에서 동작시키되, Registry와 Workspace 구조는 관리자 메뉴를 나중에 제거하거나 별도 앱으로 옮겨도 현장 앱이 흔들리지 않는 형태로 만든다.

## 4. 운영 워크플로우

현장 설치 후 앱은 아래 흐름을 기준으로 동작해야 한다.

```text
1. 배포 패키지 설치
2. admin 로그인
3. 설정 메뉴 접근
4. 기존 엑셀 데이터 컬럼 매핑
5. 로컬 DB로 기존 데이터 입력
6. 매핑 설정 저장
7. admin 로그아웃
8. 현장관리자 로그인
9. 출근 기록 생성
10. 위치/원격 접속 감지 기록
11. 현장 운영 메뉴 진입
12. 오늘 업무 데이터 입력
13. 로컬 DB 우선 저장
14. 서버 백그라운드 동기화
15. 일지 출력
16. 현장관리자 로그아웃
17. 퇴근 기록 및 미동기화 재시도
18. 미로그아웃 시 저녁 8시 자동 로그아웃
19. 로그인 대기 화면 복귀
```

로그인 화면의 기본 이름은 현장관리자 이름이어야 하며, `admin`이 자동 입력되면 안 된다.

원격 접속 여부는 관리자에게 공지하지 않고 로컬 DB에 기록한다. 현장관리자는 로그인 후 업무를 계속할 수 있어야 한다.

## 5. 사용자, 현장, 식별자 정책

시스템 연결은 이름이 아니라 id 기준으로 통일한다.

```text
site_id: UUID 문자열
member_id: UUID 문자열
local_id: 로컬 DB 내부 INTEGER id
member_name: 표시용 이름
site_name: 표시용 현장명
```

정책:

- 신규 현장 id는 UUID 문자열로 생성한다.
- 신규 회원 id도 UUID 문자열로 생성한다.
- 기존 `Date.now()` 기반 숫자 문자열 회원 id 생성은 제거한다.
- 기존 3명 회원의 id도 UUID로 전환한다.
- 회원 id 변경 시 `members`, `member_sites`, `attendance`, Google Sheets, BigQuery 참조를 함께 갱신한다.
- 이름은 표시, 검색, 로그인 힌트 용도로만 사용한다.

회원 로컬 캐시 정책:

- 회원 관리의 원본은 Google Sheets이다.
- 현장 로컬 DB에는 오프라인 로그인을 위해 필요한 회원정보만 캐시한다.
- 현장관리자 앱의 백그라운드 동기화가 Google Sheets의 전체 회원 목록을 로컬 DB에 저장하면 안 된다.
- 온라인 로그인 성공 시 해당 로그인 회원만 로컬 DB에 캐시한다.
- 관리자 설정/회원관리 화면은 Google Sheets를 직접 관리하되, 현장 운영용 로컬 캐시와 분리한다.
- admin 계정은 로그인 힌트와 현장관리자 로컬 캐시 대상에서 제외한다.

확인된 스키마 이슈:

- 코드와 초기화 스크립트 기준 BigQuery `attendance.member_id`는 `STRING`이다.
- 이미 생성된 BigQuery 실제 테이블의 타입이 `INTEGER`로 남아 있으면, 배포 전 백업 테이블 생성 후 `STRING` 스키마로 교체한다.
- 앱 코드는 BigQuery 전송 시 `member_id`를 문자열로 전송한다.

## 6. 메뉴와 워크스페이스 구조

현재 `App.jsx`가 메뉴 id를 직접 `switch`로 판단해 화면을 렌더링하는 구조는 확장에 약하다. 메뉴와 워크스페이스를 각각 등록형 구조로 분리한다.

```text
Menu Registry
├─ menuId
├─ label
├─ icon
├─ parent/children
├─ allowedRoles
├─ allowedSiteModes
├─ workspaceId
└─ displayOrder

Workspace Registry
├─ workspaceId
├─ title
├─ component
├─ layoutType
├─ helpText
├─ dataScope
├─ syncTarget
└─ requiredContext
```

원칙:

- 메뉴는 사용자가 무엇을 선택할 수 있는지만 정의한다.
- 워크스페이스는 오른쪽 화면에 무엇을 어떻게 보여줄지만 정의한다.
- 메뉴 UI가 바뀌어도 워크스페이스 정의는 유지될 수 있어야 한다.
- 워크스페이스 UI가 바뀌어도 메뉴 구조는 유지될 수 있어야 한다.
- Registry 검증으로 메뉴는 있는데 워크스페이스가 없거나, 워크스페이스는 있는데 접근 경로가 없는 상태를 잡는다.

## 7. AppContext

모든 워크스페이스는 흩어진 사용자 정보를 직접 판단하지 않고 정리된 앱 컨텍스트를 받는다.

```text
AppContext
├─ currentUser
├─ role
├─ isAdmin
├─ isFieldManager
├─ siteMode
├─ managedSites
├─ activeSiteId
├─ activeSiteName
├─ dataScope
└─ syncStatus
```

한방향/양방향은 사용자 속성으로 본다. 양방향 사용자는 하나의 앱에서 여러 현장을 관리할 수 있지만, `activeSiteId` 기준으로 데이터가 섞이지 않아야 한다.

## 8. AccountWidget

현재 사이드바 안에 들어 있는 접속자 정보 영역을 독립 위젯으로 분리한다.

```text
AccountWidget
├─ 접속자 이름
├─ 현재 현장
├─ 한방향/양방향 상태
├─ 양방향 현장 선택
├─ 비밀번호 수정
├─ 로그아웃
└─ 동기화 상태 표시
```

`AccountWidget`은 단순 표시 위젯이 아니라 앱 컨텍스트를 바꾸는 위젯이다.

```text
AccountWidget에서 activeSite 변경
→ AppContext 업데이트
→ 필요한 메뉴 재계산
→ ActiveWorkspace에 새 dataScope 전달
→ 각 화면은 해당 현장의 데이터만 조회/저장
```

이번 버전에서는 `AccountWidget`의 실제 위치를 현재 사이드바 안에 유지할 수 있다. 중요한 것은 위치를 고정하지 않고 독립 위젯으로 분리해 두는 것이다. 다음 업그레이드에서 사이드바를 접거나 숨길 경우, `AccountWidget`은 상단 바, 플로팅 패널, 또는 별도 접속자 영역으로 이동할 수 있어야 한다.

## 9. WorkspaceShell과 위젯 구조

각 화면이 직접 전체 레이아웃을 만들지 않도록 `WorkspaceShell`을 도입한다.

```text
WorkspaceShell
├─ WorkspaceToolbar
├─ WorkspaceLayout
│  ├─ filterPanel
│  ├─ mainPanel
│  ├─ sidePanel
│  └─ footerPanel
└─ WorkspaceWidgetRenderer
```

초기에는 기존 View를 `WorkspaceAdapter`로 감싸서 기능을 유지한다. 이후 패널 구조가 분명한 화면부터 위젯으로 분리한다.

이번 버전에서는 우측 워크스페이스의 기본 위치와 사용 흐름을 급격히 바꾸지 않는다. 리팩토링의 목적은 현재 배치를 즉시 바꾸는 것이 아니라, 다음 업그레이드에서 사이드바 접기/숨김 모드가 들어왔을 때 아래 배치 전환이 가능하도록 준비하는 것이다.

```text
현재 버전
└─ Sidebar 고정
   └─ Workspace는 현재 우측 영역 사용

다음 업그레이드
└─ Sidebar collapsed/hidden
   └─ Workspace는 남는 영역 또는 화면 중앙 기준으로 배치
```

우선 분리 후보:

- 대시보드: 유량/수질/재고 위젯
- 일지: 달력/조건 패널, 미리보기 패널, 출력 액션
- 슬러지: 반출 입력, 사진대지 출력, 반출관리대장 출력
- 접속자 정보: 사용자/현장/로그아웃/비밀번호 수정

슬러지 관련 구조는 하나의 업무 워크스페이스 안에 액션 위젯을 둔다.

```text
Sludge Workspace
├─ SludgeInputWidget
├─ SludgePhotoWidget
├─ SludgeLedgerExportWidget
└─ SludgePhotoReportExportWidget
```

## 10. 현장관리자 UI 입력 전략

현장관리자의 핵심 업무는 그날의 데이터를 빠르고 정확하게 입력하고 일지를 출력하는 것이다.

주요 입력:

- 유량 검침값
- 약품 사용량
- 수질분석치
- 키트 사용량
- 약품 입고량과 사진
- 슬러지 반출량과 사진
- 시설 고장/수리 이력

기본 UI는 엑셀형 그리드에 직접 입력하는 방식보다 `ListView + 입력 모달` 구조를 우선한다. 워크스페이스에는 조밀한 목록을 보여주고, 검침값/사용량/사진 같은 실제 입력은 버튼을 눌러 비주얼베이직 입력폼처럼 모달로 띄워 처리한다.

```text
워크스페이스에 ListView 표시
→ 입력 버튼 클릭
→ ToastModal/InputModal 표시
→ 큰 입력칸에서 값 입력
→ 저장
→ 모달 닫기
→ ListView와 요약 갱신
→ 일지 출력
```

화면 배치는 작은 현장 PC에서도 한 화면에 목록을 우선 보여주는 것을 기본으로 한다. `왼쪽 목록 + 오른쪽 입력폼` 배치는 넓은 화면에서 선택 가능한 레이아웃일 뿐, 기본 방향은 아니다.

```text
기본 레이아웃
└─ Workspace
   ├─ CompactListView
   ├─ ActionToolbar
   │  ├─ 검침값 입력
   │  ├─ 사용량 입력
   │  ├─ 사진 첨부
   │  └─ 일지 출력
   └─ SummaryBar

입력 시
└─ ToastModal/InputModal
   ├─ 큰 라벨
   ├─ 큰 입력칸
   ├─ 계산 미리보기
   ├─ 저장
   └─ 취소
```

엑셀형 그리드는 아래 용도로 남긴다.

- 관리자 설정
- 초기 엑셀 데이터 확인
- 대량 데이터 검토
- 고급 수정 모드

입력 모달 원칙:

- 모달은 빠르게 열리고 닫혀야 한다.
- 입력 항목은 업무별로 최소화한다.
- 글자와 입력칸은 크게 둔다.
- 저장 전 계산 결과나 재고 변화가 필요한 경우 모달 안에서 바로 보여준다.
- 저장 성공 후 ListView가 즉시 갱신되어야 한다.
- 키보드 입력과 마우스 클릭 모두 편해야 한다.
- 기존 그리드 직접 입력 흐름은 단계적으로 제거하거나 고급 모드로 이동한다.

자동 계산이 필요한 업무는 그리드 대신 계산 미리보기 위젯을 둔다.

예시:

- 유량: 전일값, 금일값, 리셋 여부, 산출 유량
- 약품: 전일 재고, 입고량, 사용량, 계산 후 재고
- 수질: 초과/이상값 강조
- 키트: 현재 재고, 사용량, 보충 필요 여부

## 11. 데이터 수정 위젯

기존 데이터 수정은 기본 입력 흐름에서 분리한다.

```text
DataCorrectionWidget
├─ 날짜/기간 선택
├─ 업무 종류 선택
├─ 기존 데이터 조회
├─ 변경 전/후 표시
├─ 수정 사유 입력
├─ 저장
└─ 수정 이력 기록
```

원칙:

- 현장관리자 기본 화면은 오늘 업무 입력에 집중한다.
- 과거 데이터 수정은 별도 위젯에서 수행한다.
- 수정 기능에는 확인 절차를 둔다.
- 수정 이력 또는 최소한 변경 전/후 로그를 남긴다.
- 권한에 따라 수정 가능 범위를 나중에 제한할 수 있어야 한다.

## 12. 동기화 구조

서버 전송 대상 업무는 업무별 동기화 모듈로 분리한다. 이 동기화의 최종 목적은 단순 백그라운드 업로드가 아니라, 현장관리자 PC의 로컬 DB가 랜섬웨어, 디스크 고장, 실수 삭제 등으로 사라졌을 때 서버 데이터를 기준으로 현장 업무 데이터를 다시 내려받아 복구할 수 있게 하는 것이다.

```text
Sync Engine
├─ flowSyncModule
├─ medicineSyncModule
├─ kitSyncModule
├─ waterQualitySyncModule
├─ facilitySyncModule
├─ sludgeSyncModule
├─ attendanceSyncModule
├─ certificateSyncModule
└─ boardSyncModule
```

각 모듈은 자기 업무 데이터만 책임진다.

```text
Sync Module
├─ getPendingItems()
├─ buildPayload()
├─ pushToServer()
├─ fetchRemoteSnapshot()
├─ restoreLocal()
├─ markSynced()
├─ rollbackPending()
└─ getStatus()
```

공통 엔진은 실행 시점과 재시도 정책만 담당한다.

```text
업무 저장
→ 로컬 DB 저장
→ pending 등록
→ 백그라운드 전송
→ 성공 시 synced 처리
→ 실패 시 pending 유지
→ 인터넷 연결 복구 시 재시도
→ 인터넷 연결이 확인되는 경우에만 1시간마다 재시도
→ 로그아웃 시 전체 재시도
```

로컬 DB가 우선이며, 서버 장애가 업무 입력을 막으면 안 된다. 인터넷 연결이 없거나 불확실한 상태에서는 BigQuery 전송을 계속 시도하지 않고 pending 상태만 유지한다. 재시도는 인터넷 연결이 확인된 상태에서 서버/BigQuery 오류가 발생했을 때 수행한다.

복구 기준:

- BigQuery는 현장 운영 데이터의 서버 백업본 역할을 한다.
- 복구 키는 `site_id + local_id + 업무별 자연키(date/type/name 등)`를 함께 고려한다.
- 현재 BigQuery 업로드가 `WRITE_APPEND` 방식이므로, 복구 시에는 `updated_at` 또는 `uploaded_at` 기준 최신 레코드를 선택해야 한다.
- `site_id`가 없는 legacy 데이터는 `site_name` fallback을 두되, 새 데이터는 `site_id` 기준을 우선한다.
- 복구는 자동 덮어쓰기가 아니라 관리자 확인 또는 명시적 복구 명령으로 수행한다.
- 이번 리팩토링에서는 복구 버튼을 만들지 않는다. 복구 UI/API는 향후 업그레이드에서 추가하되, 현재 단계에서는 모듈 인터페이스와 복구 쿼리 계약만 준비한다.
- 새 로컬 DB로 복구할 때는 서버 데이터에서 로컬 테이블을 재구성하고, 복구된 행은 `is_synced = 1`로 표시한다.
- 복구 대상은 유량, 약품, 키트, 수질, 시설관리, 출퇴근, 성적서 캐시처럼 서버에 백업 가능한 데이터이다.
- 설비이력카드처럼 로컬 전용으로 결정한 데이터는 복구 대상이 아니므로, 별도 로컬 백업 정책이 필요하다.
- 사진 파일은 BigQuery만으로 복구할 수 없으므로 Google Drive/파일 서버 URL 또는 별도 파일 백업 정책을 함께 유지해야 한다.
- 회원정보는 전체 회원 로컬 캐시 대상이 아니며, 복구 시에도 로그인 사용자 또는 관리자 앱/Google Sheets를 기준으로 필요한 회원만 다시 캐시한다.

다음 업그레이드 숙제: 미동기화 데이터 백업/복구 모듈

이번 버전에서는 구현하지 않고, 동기화 구조가 안정된 뒤 별도 업그레이드로 개발한다.

목표:

- 서버에 아직 올라가지 않은 `is_synced = 0` 데이터만 강하게 보호한다.
- 서버 동기화 완료 데이터는 BigQuery/Drive에서 다시 받을 수 있으므로 로컬에서는 캐시로 본다.
- 오프라인 중 발생한 데이터, 서버 장애 중 발생한 데이터, 동기화 실패 pending 데이터는 암호화 백업으로 보호한다.

후보 모듈:

```text
PendingBackupService
├─ collectPendingRows()
├─ collectPendingFiles()
├─ encryptBackup()
├─ writeBackupFile()
├─ pruneSyncedBackups()
└─ getBackupStatus()

ServerRecoveryService
├─ fetchLatestServerSnapshot()
├─ rebuildLocalTables()
├─ restoreDownloadedFiles()
└─ markRestoredRowsSynced()

PendingRecoveryService
├─ readEncryptedBackup()
├─ decryptBackup()
├─ restorePendingRows()
├─ restorePendingFiles()
└─ markRowsPendingForSync()
```

보안 조건:

- 백업 파일은 평문 JSON으로 남기지 않는다.
- 암호화 키/암호는 앱 코드에 하드코딩하지 않는다.
- 백업 파일 위치는 앱 데이터 폴더 내부만 고집하지 않고, 별도 보호 폴더 또는 외부 저장 위치를 검토한다.
- 백업 파일도 랜섬웨어 대상이 될 수 있으므로 보관 개수, 보관 기간, 쓰기 권한을 제한한다.
- 사진/첨부파일은 DB 행과 별도로 파일 백업 정책을 둔다.
- 복구는 자동 실행하지 않고 관리자 확인 또는 명시적 복구 버튼으로만 수행한다.

구현 순서:

1. pending 데이터 추출 범위 정의
2. 백업 파일 포맷과 암호화 방식 결정
3. pending 저장 직후 백업 생성
4. 서버 동기화 성공 후 백업 정리
5. 서버 스냅샷 복구 기능 구현
6. pending 백업 복구 기능 구현
7. 관리자 전용 앱 또는 고급 설정 화면에 복구 UI 추가
8. 랜섬웨어/DB 삭제 가정 복구 테스트

구현 단계:

```text
1. 로컬 저장 후 pending 유지
2. 업무별 push module로 BigQuery 업로드
3. 성공 시 로컬 synced 처리
4. 서버에 최신 스냅샷 조회 쿼리 준비
5. 로컬 DB 유실 시 site_id 기준으로 최신 스냅샷 다운로드
6. 로컬 테이블 재구성
7. 복구 후 전체 업무 화면 조회/일지 출력 검증
```

## 13. BigQuery와 서버 스키마 정책

앱 설치 때마다 BigQuery 테이블을 만들지 않는다. 서버는 이미 준비된 스키마를 가져야 한다.

필요한 작업:

- BigQuery 기준 스키마 정의를 한 곳으로 모은다.
- 실제 BigQuery 테이블 스키마 검증 도구를 만든다.
- 스키마 검증은 관리자 설정 또는 배포 전 체크에서 수행한다.
- 기존 테이블의 타입 불일치는 자동 추가가 아니라 별도 마이그레이션으로 처리한다.

현재 확인된 사항:

- `flow_readings`, `medicine_logs`, `water_quality`, `kit_logs`, `facility_logs`는 로컬 `id`를 BigQuery `local_id`로 보낸다.
- 로컬 `is_synced`는 BigQuery로 보내지 않는다.
- `members.password`는 BigQuery로 보내지 않는다.
- `sludge_photo_logs`는 현재 로컬 중심이며, 서버 동기화 대상 포함 여부를 결정해야 한다.
- 주요 업무 테이블은 현재 `WRITE_APPEND` 성격이므로, 중복/수정 데이터 처리 정책을 정해야 한다.

업무별로 `append`와 `merge` 정책을 명확히 결정한다.

## 14. 소통게시판 권한과 가시성 정책

소통게시판은 현장관리자와 관리자 사이의 업무 소통 채널이므로, 목록 조회, 상세 조회, 글쓰기, 댓글/답글까지 같은 가시성 규칙을 적용해야 한다.

앱 분리 전략에 따라 이번 현장관리자용 앱에서는 현장관리자용 게시판 화면을 우선 구현한다. 최고/중앙관리자용 전체 게시판 관리, 현장별 필터, 전체 공지 작성, 특정 현장/현장관리자 대상 글쓰기 화면은 관리자 전용 앱을 만들 때 구현한다. 단, 서버 가시성 정책과 데이터 스키마는 두 앱이 함께 쓸 수 있도록 미리 분리해 둔다.

다음 업그레이드 후보: 본사 양식 기반 웹 문서 작성

휴가계, 연차계, 사직서 등 본사 양식대로 작성해야 하는 문서는 일반 게시글과 섞지 않고 별도 `WebFormDocument` 또는 `FormDocument` 기능으로 분리한다. 소통게시판에서 버튼/탭으로 진입할 수는 있지만, 데이터 구조와 템플릿 관리는 독립 모듈로 둔다.

```text
소통게시판
├─ 일반 문의/공지 글
└─ 본사 양식 문서 작성
   ├─ 휴가계
   ├─ 연차계
   ├─ 사직서
   ├─ 기타 신청서
   └─ 제출/다운로드/게시글 첨부
```

현장관리자용 앱 역할:

- 양식 선택
- 입력폼 작성
- 미리보기
- PDF/엑셀/문서 출력 또는 게시글 첨부
- 자신이 작성한 제출 기록 조회

관리자 전용 앱 역할:

- 양식 템플릿 등록/수정
- 제출 문서 전체 조회
- 현장/작성자별 필터
- 문서 다운로드/보관
- 승인/반려 워크플로우는 추후 확장

이번 리팩토링에서는 이 기능을 구현하지 않고, 게시판 가시성/권한 구조가 향후 `FormDocument`와 연결될 수 있도록 게시글 첨부/문서 참조 확장 포인트만 남긴다.

목표 정책:

```text
현장관리자
├─ 전체 공지 조회 가능
├─ 자신 또는 자신의 현장과 관련된 게시글 조회 가능
├─ 자신이 작성한 글은 자신과 중앙/최고관리자만 볼 수 있음
├─ 자신이 볼 수 있는 게시글의 댓글/답글만 조회 가능
└─ 글 작성 시 별도 대상 선택 없이 관리자 대상 글로 저장

중앙/최고관리자
├─ 모든 현장의 모든 게시글 조회 가능
├─ 현장별 필터링 가능
├─ 전체 현장관리자 대상 공지 작성 가능
├─ 특정 현장관리자 또는 특정 현장 대상 글 작성 가능
└─ 모든 게시글의 댓글/답글 조회 가능

이번 현장관리자용 앱 구현 범위
├─ 현장관리자 목록 조회
├─ 현장관리자 상세 조회
├─ 현장관리자 글쓰기(admin_only 기본 scope)
├─ 현장관리자 댓글/답글 작성
└─ 현장관리자에게 허용된 글/댓글만 표시

관리자 전용 앱 이관 범위
├─ 전체 게시글 관리
├─ 현장별/현장관리자별 필터
├─ 전체 공지 작성
├─ 특정 현장 대상 글 작성
├─ 특정 현장관리자 대상 글 작성
└─ 관리자용 댓글/삭제/관리 화면
```

게시글에는 표시용 이름 외에 가시성 판단용 id를 저장해야 한다.

```text
posts
├─ id
├─ author_id
├─ author_name
├─ author_role
├─ author_site_id
├─ author_site_name
├─ visibility_scope: all | admin_only | site | member
├─ target_site_id
├─ target_site_name
├─ target_member_id
├─ target_member_name
├─ title
├─ content
├─ is_notice
├─ parent_id
├─ is_deleted
├─ created_at
└─ updated_at

comments
├─ id
├─ post_id
├─ parent_id
├─ author_id
├─ author_name
├─ content
├─ is_deleted
└─ created_at
```

현재 코드에서 확인된 보완 필요사항:

- `BoardView`와 서버 서비스가 `admin`만 관리자처럼 처리하고 있어 `group_admin`이 관리자 권한으로 동작하지 않을 수 있다.
- 현장관리자 역할 값이 실제 앱에서는 `user`인데 게시판 코드 일부 주석/기본값은 `manager`를 사용한다. 역할 명칭을 통일해야 한다.
- 현장관리자 작성 글이 `author_site` 기준으로 노출되어 같은 현장 다른 사용자에게 보일 가능성이 있다. 요구사항이 "자신과 중앙/최고관리자만"이라면 `author_id` 또는 `visibility_scope=admin_only` 기준으로 바꿔야 한다.
- 게시글 목록에는 가시성 필터가 있으나, 단건 조회와 댓글 조회에는 동일한 가시성 검증이 부족하다.
- 댓글 삭제가 작성자/관리자 권한을 확인하지 않고 삭제될 수 있다.
- 프론트는 댓글 답글 `parent_id`를 다루지만, 서버의 댓글 생성과 BigQuery `comments` 스키마에는 `parent_id`가 반영되어 있지 않다.
- 관리자 글쓰기 화면은 현장명을 직접 입력하는 방식이라 현장 선택 실수가 생길 수 있다. 이 화면은 관리자 전용 앱 이관 시 현장/현장관리자 선택 UI로 새로 구현한다.
- 관리자 목록 화면에 현장별 필터가 없다. 이 기능은 관리자 전용 앱 이관 범위로 둔다.
- 공지, 전체 공지, 특정 현장/특정 현장관리자 대상 글의 의미가 코드상 명확히 분리되어 있지 않다.

리팩토링 방향:

- 게시판 전용 `BoardVisibilityPolicy`를 만든다.
- 프론트와 백엔드에서 같은 역할 판단 규칙을 사용한다.
- `admin`, `group_admin`은 관리자 계열로 묶는다.
- `user`는 현장관리자로 통일한다.
- 목록 조회, 단건 조회, 댓글 조회, 댓글 작성, 삭제에 모두 가시성 검증을 적용한다.
- 현장관리자용 앱에서는 관리자 대상 선택 UI를 노출하지 않는다.
- 관리자 글쓰기의 `전체 공지`, `특정 현장`, `특정 현장관리자` 선택은 관리자 전용 앱에서 구현한다.
- 현장관리자 글쓰기는 기본적으로 `visibility_scope=admin_only` 또는 요구 정책에 맞는 전용 scope로 저장한다.
- 댓글/답글도 원글을 볼 수 있는 사용자만 볼 수 있고 작성할 수 있게 한다.
- BigQuery `posts`, `comments` 스키마에 필요한 가시성 컬럼을 추가하거나 별도 마이그레이션 계획을 세운다.

검증 항목:

- 현장관리자는 전체 공지와 자신에게 허용된 글만 목록에서 본다.
- 현장관리자는 URL/id를 직접 알아도 허용되지 않은 글 상세를 볼 수 없다.
- 현장관리자는 허용된 글의 댓글/답글만 볼 수 있다.
- 현장관리자 작성 글은 다른 무관한 현장관리자에게 보이지 않는다.
- 중앙/최고관리자 전체 조회/필터/대상 선택 글쓰기는 관리자 전용 앱 구현 시 검증한다.
- 댓글 삭제는 작성자 또는 관리자만 가능하다.

## 15. 시설관리 현장별 데이터 격리

시설관리도 게시판과 비슷하게 현장별 가시성 정책이 필요하다. 다만 시설관리는 게시판이 아니므로 댓글/답글 구조는 없다.

목표 정책:

```text
현장관리자
├─ 자신의 현장 시설관리 기록만 조회
├─ 자신의 현장 시설관리 기록만 작성
├─ 자신의 현장 시설관리 기록만 수정
└─ 자신의 현장 시설관리 기록만 삭제 또는 삭제 요청

중앙/최고관리자
├─ 모든 현장 시설관리 기록 조회
├─ 현장별 필터링 가능
├─ 특정 현장 시설관리 기록 작성 가능
├─ 특정 현장 시설관리 기록 수정 가능
└─ 특정 현장 시설관리 기록 삭제 가능
```

현재 코드에서 확인된 보완 필요사항:

- `facilityRoutes.cjs`의 목록 API는 `site_id` 쿼리 필터를 지원한다.
- `FacilityModel.fetchAll()`과 `useFacilityViewModel()`은 현재 `activeSiteId`를 전달하지 않는다.
- 현장관리자도 필터 없이 호출하면 전체 현장 시설 기록을 볼 수 있는 위험이 있다.
- 등록 시에는 `getCurrentRecordMetadata()`로 현장 메타데이터를 저장하지만, 조회/수정/삭제의 현장 범위 검증은 부족하다.
- 수정/삭제 API는 `id`만 기준으로 처리하므로 다른 현장 데이터도 id를 알면 변경될 수 있다.
- 로컬 `facility_logs`와 BigQuery `facility_logs`에는 이미 `company`, `price` 컬럼이 있으나 현재 프론트 입력/라우트 저장 흐름에서는 거의 사용하지 않는다.
- 견적 내용, 업체 담당자, 연락처, 첨부파일, 작업 상태 같은 향후 확장 데이터는 단순 컬럼 추가만으로는 길어질 수 있으므로 확장 방식을 미리 정해야 한다.

리팩토링 방향:

- 시설관리 API도 `AppContext.dataScope` 기준으로 조회한다.
- 현장관리자 요청은 서버에서 반드시 `site_id = activeSiteId` 조건을 강제한다.
- 관리자 계열은 전체 조회와 현장별 필터를 모두 지원한다.
- 수정/삭제 시에도 대상 레코드의 `site_id`가 현재 사용자 권한 범위에 있는지 확인한다.
- 프론트에서만 필터링하지 않고 서버 라우트에서 최종 검증한다.
- 시설관리 워크스페이스에는 관리자용 현장 필터와 현장관리자용 고정 현장 표시를 분리한다.
- 현장관리자용 기본 입력 화면은 핵심 항목만 유지하고, 업체/견적 상세는 별도 상세 입력 모달 또는 고급 위젯으로 분리한다.
- 이미 있는 `company`, `price`는 유지하되, 다음 확장 시 실제 UI/라우트/BigQuery 매핑에 연결한다.

시설관리 확장 후보 컬럼:

```text
facility_logs
├─ vendor_name            업체명
├─ vendor_contact_name    업체 담당자
├─ vendor_phone           업체 연락처
├─ estimate_title         견적 제목
├─ estimate_amount        견적 금액
├─ actual_amount          실제 수리비
├─ work_status            접수 | 견적 | 작업중 | 완료 | 보류
├─ requested_at           요청일시
├─ completed_at           완료일시
├─ attachment_json        견적서/사진/문서 파일 메타데이터
└─ extra_json             현장별 임시 확장 데이터
```

확장 원칙:

- 자주 검색/필터링할 값은 컬럼으로 둔다.
- 현장마다 다를 수 있는 부가 정보는 `extra_json` 같은 확장 JSON에 둔다.
- 견적서, 거래명세서, 사진은 DB 컬럼이 아니라 파일 저장소 경로/메타데이터로 관리한다.
- 금액 컬럼은 `INTEGER`만 고집하지 말고 향후 세금/공급가/부가세 분리 가능성을 고려한다.
- BigQuery 복구와 분석 대상이 되는 값은 로컬 DB와 BigQuery 스키마를 함께 설계한다.

검증 항목:

- 현장관리자는 다른 현장 시설 기록을 볼 수 없다.
- 현장관리자는 다른 현장 시설 기록을 수정/삭제할 수 없다.
- 현장관리자가 작성한 시설 기록은 자신의 `activeSiteId`로 저장된다.
- 중앙/최고관리자는 전체 시설 기록을 볼 수 있다.
- 중앙/최고관리자는 현장별 필터링을 할 수 있다.
- 양방향 사용자가 현장을 전환하면 시설관리 목록/저장 대상도 함께 전환된다.

설비이력카드:

- 설비이력카드는 향후 구현 대상이다.
- 설비이력카드는 현장 내부 관리용 데이터로 보고 서버 BigQuery 동기화 대상에 포함하지 않는다.
- 설비이력카드는 시설관리 워크스페이스 안의 별도 위젯 또는 하위 워크스페이스로 추가할 수 있게 구조만 열어둔다.
- 설비이력카드가 고장/수리이력과 연결되더라도, 서버 동기화 대상은 `facility_logs` 같은 시설관리 이력 데이터로 제한한다.
- 설비이력카드 사진/사양/설치일 등은 로컬 DB와 로컬 파일 저장소 기준으로 관리한다.

장비이력카드 후보 테이블:

```text
equipment_cards
├─ id                      로컬 INTEGER 또는 UUID
├─ site_id
├─ equipment_code          장비 코드
├─ equipment_name          장비명
├─ category                장비 분류
├─ location                설치 위치
├─ manufacturer            제조사
├─ model_name              모델명
├─ serial_no               제조번호
├─ specification           주요 사양
├─ installed_at            설치일
├─ warranty_until          보증 만료일
├─ vendor_name             관리/납품 업체
├─ vendor_phone            업체 연락처
├─ photo_json              장비 사진 메타데이터
```

시설관리 이력과 장비이력카드 연결:

- `facility_logs.equipment_id` 또는 `equipment_code`로 장비이력카드를 참조한다.
- 기존 자유 입력 `facility_name`은 유지하되, 장비 카드가 있는 경우 장비 선택으로 자동 채운다.
- 장비이력카드는 서버 동기화 대상이 아니므로 로컬 백업/복구 모듈의 다음 업그레이드 과제에 포함한다.

운영 원칙:

- 법정/본사 양식은 템플릿 파일로 관리하고, 앱은 현장명/일자/작업명/업체명 등 기본값을 자동 채운다.
- 서명 자체를 전자서명으로 대체하려고 하지 않는다. 종이 출력과 수기 서명을 기준으로 하되, 앱은 이미지 증빙을 보관한다.
- 서명본 이미지는 작업 전/중/후 사진과 구분해 `signed_form` 성격으로 관리한다.
- 이미지 원본은 로컬 파일 저장소에 저장하고, DB에는 경로와 메타데이터만 저장한다.
- 서버 동기화 여부는 다음 업그레이드에서 결정하되, 오프라인 중 촬영한 서명본 이미지는 `PendingBackupService` 보호 대상에 포함한다.

UI 방향:

- 시설관리 목록은 요약만 보여준다.
- 보수공사 등록/상세는 큰 모달 또는 별도 워크스페이스로 연다.
- 입력 모달은 `기본정보`, `업체/견적`, `투입인원`, `사진`, `안전관리`, `첨부문서` 탭으로 나눈다.
- 안전관리 탭에는 안전 양식 출력과 서명본 이미지 첨부 영역을 둔다.
- 사진은 작업 전/중/후/안전/허가서/가스측정 단계별로 여러 장 첨부할 수 있게 한다.
- 안전관리 항목은 공사 종류에 따라 필수 여부를 다르게 적용할 수 있게 한다.

동기화/백업 정책:

- 보수공사 상세와 안전관리 데이터는 `facility_logs`와 연결되는 서버 동기화 후보이다.
- 사진/허가서/문서는 BigQuery에 직접 저장하지 않고 파일 저장소 경로와 메타데이터만 저장한다.
- 오프라인 중 촬영/작성된 사진과 안전문서는 다음 업그레이드의 `PendingBackupService` 보호 대상에 포함한다.
- 장비이력카드 자체는 로컬 전용으로 유지하더라도, 보수공사와 안전관리 이력은 서버 복구 대상에 포함할지 별도 결정한다.

## 16. 성적서 워크스페이스와 리스트 위젯

성적서는 서버(BigQuery/Drive)에서 내려받을 수 있는 자료이며, 현장별 가시성 정책과 월별 조회가 중요하다. 성적서 목록 화면은 독립적인 `CertificateListWidget`으로 분리한다.

장기적으로 성적서 기능은 현장관리자용 앱과 관리자 전용 앱의 책임을 분리한다. 현장관리자용 앱은 조회와 다운로드만 담당하고, 관리자 전용 앱은 ZIP 업로드와 전체 조회/관리 기능을 담당한다.

목표 정책:

```text
현장관리자
├─ 자신의 현장 성적서만 조회
├─ 월별 필터링 가능
└─ 성적서 다운로드 가능

중앙/최고관리자
├─ 전체 현장 성적서 조회
├─ 현장별 필터링 가능
├─ 월별 필터링 가능
├─ 성적서 다운로드 가능
└─ 성적서 ZIP 업로드 가능
```

분리 후 목표:

```text
현장관리자용 앱
└─ 성적서 조회/다운로드

관리자 전용 앱
├─ 성적서 ZIP 업로드
├─ 전체 성적서 조회
├─ 현장별/월별 필터링
└─ 필요 시 성적서 삭제/재처리
```

위젯 구조:

```text
CertificateWorkspace
├─ CertificateFilterWidget
│  ├─ 년/월 선택
│  └─ 관리자용 현장 선택(분리 전까지만 유지)
├─ CertificateListWidget
│  ├─ 성적서 목록
│  ├─ 선택 상태
│  └─ 빈 목록/로딩 상태
├─ CertificateActionWidget
│  ├─ 다운로드
│  └─ 관리자용 ZIP 업로드(관리자 앱 이관 대상)
└─ CertificateBatchProgressWidget
   └─ ZIP 업로드 진행 상태(관리자 앱 이관 대상)
```

현재 코드에서 확인된 사항:

- `CertificateView`는 이미 리스트 중심 화면과 년/월 선택, 관리자용 현장 선택을 갖고 있다.
- `useCertificateViewModel`은 관리자 계열 역할을 `PRIVILEGED_ROLES`로 구분한다.
- 서버 `certificateRoutes.cjs`는 현장관리자(`user`)의 타 현장 성적서 조회를 차단하는 로직을 일부 갖고 있다.
- 현재 필터는 주로 `siteName` 기준이므로, 장기적으로는 `site_id` 기준을 우선하고 `site_name`은 표시/호환용으로 사용해야 한다.
- 현장관리자는 자신의 현장 성적서만 보도록 서버 검증을 유지해야 하며, 프론트 필터만 믿으면 안 된다.

리팩토링 방향:

- 성적서 목록 UI를 `CertificateListWidget`으로 분리한다.
- 년/월 필터와 관리자용 현장 필터를 `CertificateFilterWidget`으로 분리한다.
- 다운로드/업로드 액션을 `CertificateActionWidget`으로 분리한다.
- 현장관리자는 현장 선택 UI를 보지 않고 자신의 `activeSiteId`/현장명 기준으로만 조회한다.
- 중앙/최고관리자는 전체/현장별 필터를 사용할 수 있다.
- 조회와 다운로드 API 모두 서버에서 권한과 현장 범위를 검증한다.
- 가능하면 성적서 가시성 기준을 `site_id`로 전환하고, 파일명/기존 데이터 호환을 위해 `site_name` fallback을 둔다.
- 현장관리자용 최종 앱에서는 인쇄 버튼을 기본 기능에서 제외한다. 필요 시 사용자가 다운로드한 파일을 운영체제/뷰어에서 인쇄한다.
- ZIP 업로드와 전체 성적서 관리는 관리자 전용 앱 이관 대상임을 코드 경계와 Registry 메타데이터에 표시한다.

검증 항목:

- 현장관리자는 자신의 현장 성적서만 목록에서 본다.
- 현장관리자는 월별 필터링을 할 수 있다.
- 현장관리자는 다른 현장 성적서를 URL/id로 직접 요청해도 받을 수 없다.
- 중앙/최고관리자는 전체 성적서 목록을 볼 수 있다.
- 중앙/최고관리자는 현장별 필터링을 할 수 있다.
- 중앙/최고관리자는 월별 필터링을 할 수 있다.
- 다운로드는 권한이 허용된 성적서에 대해서만 동작한다.
- ZIP 업로드는 관리자 계열만 가능하다.

- 임시 코드와 장기 유지 코드를 구분한다.
- "나중에 혹시 쓸지도 모르는 코드"는 남기지 않고, 나중에 반드시 알아야 하는 의도는 문서와 주석으로 남긴다.

- 메뉴 상수와 화면 연결 불일치

## 18. 리팩토링 안전장치

린트 오류가 없다는 것만으로 리팩토링이 완료된 것으로 보지 않는다. 리팩토링은 기존 워크플로우와 기능 동작이 유지되는지까지 확인해야 완료된다.

원칙:

- 각 단계는 작게 나누어 구현한다.
- 단계별 구현 후 린트와 핵심 기능을 함께 확인한다.
- 기존 기능과 다르게 동작하는 부분이 발견되면 즉시 수정 항목으로 기록한다.
- 구조 변경 중 임시로 기능을 끊어야 하는 경우, 끊긴 기능과 복구 조건을 문서에 남긴다.
- 기존 워크플로우와 다른 동작이 의도된 변경인지, 회귀 버그인지 구분한다.
- 의도된 변경이면 계획 문서와 검증 체크리스트를 함께 수정한다.
- 회귀 버그이면 릴리즈 전에 반드시 수정한다.

구현 중 반복 체크:

```text
코드 변경
→ 린트 확인
→ 해당 화면 수동 실행 확인
→ 관련 워크플로우 확인
→ 회귀/수정사항 기록
→ 다음 단계 진행
```

특히 아래 기능은 리팩토링 중 깨지기 쉬우므로 매 단계 확인한다.

- 로그인 힌트가 admin으로 돌아가지 않는지
- admin/현장관리자 메뉴 노출이 뒤바뀌지 않는지
- 현장관리자 로그인 시 출근 기록이 생성되는지
- 로그아웃 시 퇴근 기록이 생성되는지
- 자동 로그아웃 타이머가 유지되는지
- 저장 시 로컬 DB 우선 저장이 유지되는지
- 동기화 실패가 입력 기능을 막지 않는지
- 일지 출력물이 기존 데이터 기준과 다르게 생성되지 않는지
- 양방향 현장 전환 시 `activeSiteId`가 모든 조회/저장에 반영되는지
- 소통게시판에서 현장관리자/관리자 가시성 규칙이 깨지지 않는지
- 시설관리에서 현장별 데이터 격리가 깨지지 않는지
- 성적서에서 현장별 조회/다운로드 권한이 깨지지 않는지

## 19. 보안 및 배포 산출물 관리

현재 작업 트리에는 민감 파일과 릴리즈 산출물이 포함될 수 있다.

정리 원칙:

- `client_secret*.json` 같은 민감 파일은 커밋하지 않는다.
- `release/`, `release-temp/` 산출물은 소스 변경과 분리한다.
- 배포에 필요한 키/설정 파일은 보관 위치와 패키징 포함 여부를 명확히 한다.
- GitHub 릴리즈에 올릴 파일과 저장소에 커밋할 파일을 구분한다.
- 배포 전 `.gitignore`와 패키징 설정을 점검한다.

12단계 적용 결과:

- `.gitignore`에 OAuth client secret 파일, `release/`, `release-temp/`, `*.blockmap`을 추가해 민감 파일과 패키징 산출물이 소스 변경에 섞이지 않도록 했다.
- 현재 구조에서 실제 사용처가 없는 레거시 Google Drive JSON 동기화 시뮬레이션 파일 `src/services/DriveSyncService.js`를 제거했다.
- `useAttendanceViewModel.js`의 미사용 `DriveSyncService` import를 제거했다.
- 회원 전체 로컬 캐시 정책과 맞지 않고 호출되지 않는 `authRoutes.cjs`의 `syncLocalMembers` 헬퍼를 제거했다.
- 보존 대상인 마이그레이션 스크립트, BigQuery 스키마 스크립트, 로컬 DB migration 코드는 릴리즈 전후 추적과 복구 가능성을 위해 삭제하지 않는다.

`WorkspaceAdapter`는 기존 화면을 깨지 않고 공통 메타데이터를 붙이는 얇은 껍데기로 둔다. 이후 위젯화는 화면 내부를 한 번에 갈아엎지 않고, 데이터 흐름과 사용자 동작이 분리되는 지점부터 순서대로 진행한다.

우선순위:

```text
1. 대시보드
2. 일지 작성/출력
3. 슬러지 사진대지/반출관리대장
4. 유량/약품/키트/수질 입력 화면
5. 시설관리 고급 입력 모달
```

대시보드 위젯화:

- 현재 이미 `FlowTrendWidget`, `WaterQualityWidget`, `InventoryLevelWidget`로 일부 분리되어 있다.
- 후속 작업은 각 위젯의 공통 껍데기(`DashboardWidgetFrame`)와 접기/펼치기 상태 저장 방식을 통일하는 것이다.
- 대시보드 전체는 현장관리자 업무 상태 요약에 집중하고, 관리자용 통계/전체 현장 비교는 관리자 전용 앱으로 이관한다.

```text
DashboardWorkspace
├─ DashboardWidgetFrame
│  ├─ title
│  ├─ collapse/expand
│  └─ status/loading
├─ FlowTrendWidget
├─ WaterQualityWidget
└─ InventoryLevelWidget
```

일지 작성/출력 위젯화:

- 현재 일지는 `DailyLogView`와 `DailyLogStatusDashboard`에 조회 조건, 미리보기, 출력 액션, 상태 표시가 섞여 있다.
- 현장관리자에게 중요한 것은 날짜/서식 선택, 미리보기 확인, 출력 파일 생성이다.
- 후속 위젯화는 출력 대상 선택과 미리보기/상태/액션을 분리한다.


공통 위젯화 원칙:

- ViewModel의 데이터/핸들러 계약을 먼저 안정화하고, UI 위젯은 그 계약을 props로 받는다.
- 위젯은 직접 API를 호출하지 않는다.
- 현장관리자 기본 화면은 조회 목록과 큰 입력 모달/폼 중심으로 단순화한다.
- 출력/동기화/업로드처럼 시간이 걸리는 작업은 별도 진행 상태 위젯으로 분리한다.
- 관리자 앱 이관 대상 기능은 위젯 이름과 Registry 메타데이터에 표시한다.
- 이번 버전에서는 대규모 UI 교체보다 위젯 분리 후보와 연결 지점을 정리하는 데 집중한다.

## 21. 운영 워크플로우 검증 체크리스트

### 설치 및 관리자 설정

- 배포 패키지 설치 가능
- 앱 실행 가능
- admin 로그인 가능
- admin에게 설정 메뉴가 보임
- 현장 기본 설정 저장 가능
- 기존 엑셀 데이터 매핑 가능
- 매핑 결과가 로컬 DB에 입력됨
- 매핑 설정이 저장됨
- admin 로그아웃 가능

### 현장관리자 로그인

- 로그인 대기 화면에 admin이 자동 입력되지 않음
- 현장관리자 이름이 자동 입력됨
- 현장관리자 로그인 가능
- 로그인 시 출근 기록 생성
- 위치 일치 여부 기록
- 원격 접속 감지 결과 기록
- 현장관리자에게 설정 메뉴가 보이지 않음
- 현장 운영 메뉴만 보임

### 현장 업무 입력

- 유량 검침값 입력 가능
- 약품 사용량 입력 가능
- 수질분석치 입력 가능
- 키트 사용량 입력 가능
- 약품 입고량과 사진 입력 가능
- 슬러지 반출량과 사진 입력 가능
- 시설 고장/수리 이력 입력 가능
- 저장 시 로컬 DB에 먼저 저장됨
- 저장 후 백그라운드 동기화가 요청됨

### 일지 출력

- 일일업무일지 출력 가능
- 수질분석일지 출력 가능
- 약품관리대장 출력 가능
- 약품입고일지 출력 가능
- 슬러지사진대지 출력 가능
- 슬러지반출관리대장 출력 가능
- 출력물에 입력 데이터가 정확히 반영됨

### 동기화

- 서버 정상 시 pending 데이터가 전송됨
- 서버 장애 시 로컬 저장은 유지됨
- 서버 장애 시 pending 상태가 유지됨
- 온라인 복구 시 재시도됨
- 인터넷 연결이 확인되는 경우에만 1시간 주기 재시도됨
- 로그아웃 시에도 인터넷 연결이 확인될 때만 미동기화 전체 재시도됨
- 동기화 실패가 현장 업무 입력을 막지 않음
- 로컬 DB 유실 시 `site_id` 기준 서버 스냅샷으로 주요 업무 데이터를 재구성할 수 있음
- 복구된 데이터는 중복 없이 최신본 기준으로 로컬 DB에 들어감
- 사진/첨부파일은 서버 URL 또는 별도 파일 백업 정책으로 복구 가능 여부가 표시됨

### 출퇴근과 자동 로그아웃

- 현장관리자 로그아웃 시 퇴근 기록 생성
- 퇴근 기록 동기화 시도
- 현장관리자가 로그아웃하지 않아도 저녁 8시에 자동 로그아웃
- 자동 로그아웃 시 퇴근 기록 생성
- 자동 로그아웃 후 로그인 대기 화면으로 복귀

- 현장관리자는 자신에게 허용된 게시글만 볼 수 있음

- 중앙/최고관리자는 현장별로 게시글을 필터링할 수 있음
- 중앙/최고관리자는 전체 공지를 작성할 수 있음

- 댓글/답글 삭제 권한이 작성자 또는 관리자 계열로 제한됨

### 시설관리

- 현장관리자는 자신의 현장 시설 기록만 볼 수 있음
- 현장관리자는 다른 현장 시설 기록을 볼 수 없음
- 현장관리자는 자신의 현장 시설 기록만 작성/수정/삭제할 수 있음
- 중앙/최고관리자는 모든 현장 시설 기록을 볼 수 있음
- 중앙/최고관리자는 현장별 필터링을 할 수 있음
- 양방향 현장 전환 시 시설관리 목록과 저장 대상 현장이 함께 바뀜

### 성적서

- 현장관리자는 자신의 현장 성적서만 볼 수 있음
- 현장관리자는 월별 필터링을 할 수 있음
- 현장관리자는 다른 현장 성적서를 다운로드할 수 없음
- 중앙/최고관리자는 전체 성적서를 볼 수 있음
- 중앙/최고관리자는 현장별 필터링을 할 수 있음
- 중앙/최고관리자는 월별 필터링을 할 수 있음

- 성적서 ZIP 업로드는 분리 전 현재 앱에서는 관리자 계열만 가능하며, 이후 관리자 전용 앱으로 이관함

## 22. 최종 수정 루프

전체 워크플로우 검증 중 문제가 발견되면 릴리즈로 넘어가지 않는다.

처리 순서:

```text
문제 발견
→ 원인 분류
   - 린트/문법 문제
   - 구조 변경으로 인한 연결 문제
   - 기존 기능 회귀
   - 계획과 실제 업무 흐름 불일치
   - BigQuery/Google Sheets/로컬 DB 계약 불일치
→ 수정
→ 해당 기능 재검증
→ 전체 워크플로우 중 영향 구간 재검증
→ 체크리스트 갱신
```

완료 조건:

- 린트 오류 없음
- 주요 화면 진입 가능
- admin 설정 워크플로우 정상
- 현장관리자 운영 워크플로우 정상
- 로컬 저장과 동기화 재시도 정상
- 일지 출력 정상
- 릴리즈 산출물 생성 전 미해결 회귀 없음

## 23. 릴리즈 계획

리팩토링과 검증이 끝나면 최종 배포 패키지를 만든다.

현재는 실제 현장 설치 전이므로 GitHub의 최초 릴리즈 asset을 교체한다. 이 패키지를 진짜 최초 현장 배포본으로 본다.

릴리즈 전 체크리스트:

- git 상태 확인
- 민감 파일 제외 확인
- 릴리즈 산출물 관리 확인
- 앱 버전 확인
- Windows installer 빌드
- 설치 테스트
- admin 초기 설정 테스트
- 현장관리자 로그인 테스트
- 업무 입력/저장 테스트
- 일지 출력 테스트
- BigQuery 동기화 테스트
- 기존 GitHub 최초 릴리즈 asset 교체
- 교체된 설치 파일로 재설치 테스트

선호 릴리즈 방식:

```text
기존 v1.0.0 릴리즈 asset 교체
→ 현장 설치 전이므로 가능
→ 검증 완료 패키지를 최초 배포본으로 사용
```

## 24. 외부 아키텍처 참고

이번 리팩토링에는 외부 프레임워크를 크게 도입하지 않는다. 대신 공개적으로 검증된 패턴만 현재 구조에 맞게 반영한다.

- Local-first / Offline-first: 로컬 DB 우선, 서버 동기화는 백그라운드
- React-admin식 Resource 구조: 메뉴/권한/데이터/레이아웃 분리
- Widget Registry 패턴: 워크스페이스와 내부 위젯을 등록형으로 관리

현재 앱은 이미 Electron, Express, better-sqlite3, React MVVM 구조를 갖고 있으므로, 대형 프레임워크 도입보다 기존 구조를 정리하고 명확한 계약을 추가하는 방향이 맞다.
