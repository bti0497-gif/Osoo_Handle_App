## 1. 🔴 문제 1: `.cjs` 파일 인코딩 깨짐 (가장 심각)

### 현상
```
[server:stderr] SyntaxError: Invalid or unexpected token
    at ... waterQualityRoutes.cjs:12
    nh3_n: '?锟诫?锟届晞?锟届??,
           ^^^^^^^^^^^^^

[server:stderr] SyntaxError: Unexpected token ')'
    at ... kitRoutes.cjs:218
    });

[server:stderr] SyntaxError: Invalid or unexpected token
    at ... settingsRoutes.cjs:46
    const defaultQntechPhotoRoot = path.join(appDataPath, '?锟届甏€锟?, '?锟届攵勳劃');
```

### 원인
**모든 `.cjs` 파일이 CP949(ANSI) 인코딩으로 저장**되어 있음
- Electron Node.js는 **UTF-8**로 읽음 → 한글 깨져서 **SyntaxError**

### 영향 파일
`server/**/*.cjs` 전체 (routes 17개, services 22개, scripts 4개, cron 1개, utils 1개 = 약 45개)

### 해결책
```powershell
Get-ChildItem -Path "server" -Recurse -Filter "*.cjs" | ForEach-Object {
    $content = [System.IO.File]::ReadAllText($_.FullName, [System.Text.Encoding]::GetEncoding(949))
    [System.IO.File]::WriteAllText($_.FullName, $content, [System.Text.Encoding]::UTF8)
}
```

## 2. 🟠 문제 2: better-sqlite3 네이티브 모듈 버전 불일치 (서버 전체 중단)

### 현상 (첫 번째 실행 시 로그)
```
NODE_MODULE_VERSION 127 (개발 빌드) → NODE_MODULE_VERSION 143 (Electron 실행)
Error: The module '...\better_sqlite3.node' was compiled against a different Node.js version
```

이 에러가 나면 `registerLazyApplication()` 전체가 `throw`되어 서버가 DB 없이 시작됨.
→ 워치독(`start.cjs`)이 재시작, DB 로드 시도 → 실패 반복

### 원인
- `npm install` 시 개발 환경 Node.js 버전으로 `better-sqlite3.node`가 C++ 컴파일됨
- Electron 내장 Node.js는 더 높은 ABI 버전이라 로드 실패

### 해결책: `electron-builder` rebuild 설정

#### 권장: `npmRebuild: true` (간단하고 안정적)
```javascript
// electron-builder.config.js
module.exports = {
  // ... 기존 설정 ...
  
  // electron-builder가 패키징 시 native 모듈을 Electron Node.js에 맞춰 rebuild
  // buildDependenciesFromSource와 달리 사전 빌드된 바이너리나 시스템 도구 활용
  npmRebuild: true,
};
```

#### 대안: `buildDependenciesFromSource: true` (더 강력하지만 조건 필요)
- 모든 native 의존성을 **C++ 소스부터 재컴파일**
- **Visual C++ Build Tools** 또는 **Windows SDK**가 설치되어 있어야 함
- 설치 환경이 없다면 빌드 실패

|           | npmRebuild              | buildDependenciesFromSource |
|-----------|------------------------|-----------------------------|
| 원리      | npm rebuild와 동일      | 소스부터 전체 재컴파일       |
| 필수 도구 | node-gyp (npm 내장)     | Visual C++ Build Tools 필요  |
| 속도      | 빠름                    | 느림                         |
| 안정성    | 대부분의 경우 충분       | 더 깐깐하지만 실패 위험 있음 |

> **추천**: `npmRebuild: true`로 먼저 시도. 실패 시 `buildDependenciesFromSource: true` 고려.

## 5. 🔧 구조 개선: 라우트 레지스트리 도입 (아키텍처 개선)

### 5.1. 현재 문제점

`server/index.cjs` 내 라우트 등록 코드가 **5곳으로 분산**되어 있음:

| 위치 | 역할 | 찾기 어려움 |
|------|------|-------------|
| ① registerLazyApplication() 이전 | `BIGQUERY_IMMEDIATE_SYNC_PREFIXES` 배열 | 어떤 prefix가 있는지 별도 관리 |
| ② registerLazyApplication() 상단 | Tier 0: `app.use('/api/auth', ...)` | 즉시 등록 |
| ③ registerLazyApplication() 중반 | Tier 1: `tier1RouteConfigs` 배열 + IIFE | preload 대상 |
| ④ registerLazyApplication() 후반 | Tier 2: `makeLazy()` 호출 11개 | 첫 요청 시 로드 |
| ⑤ preload-trigger 핸들러 | Tier 1을 200ms 간격 수동 순회 | tier1RouteConfigs와 중복 |

### 5.3. `server/index.cjs` 리팩터 후 makeLazy 변경 사항

**주의**: 현재 `index.cjs` 파일 상단에 이미 `makeLazy` 함수가 있습니다.  
리팩터 시 **기존 `makeLazy` 함수를 아래처럼 교체**합니다. `registerLazyApplication()` 내부에 재정의하는 것이 아니라, **파일 스코프 수준에서** 기존 함수를 대체합니다.

```javascript
// server/index.cjs — 기존 makeLazy 함수를 아래로 교체
// (registerLazyApplication() 밖, 파일 상단)

// BEFORE (현재)
function makeLazy(modulePath, ...args) {
  let router = null;
  return (req, res, next) => {
    if (!router) {
      console.log(`[Lazy] Tier2 첫 요청 — ${path.basename(modulePath)}`);
      router = require(modulePath)(...args);
    }
    router(req, res, next);
  };
}

// AFTER (에러 처리 + 무한 재시도 방지)
function makeLazy(modulePath, ...args) {
  let router = null;
  let loadError = null;
  return (req, res, next) => {
    if (loadError) {
      return res.status(500).json({ ok: false, error: `Module previously failed: ${modulePath}: ${loadError}` });
    }
    if (!router) {
      try {
        router = require(modulePath)(...args);
      } catch (e) {
        loadError = e.message;
        console.error(`[Lazy] Tier2 로드 실패 ${modulePath}:`, e.message);
        return res.status(500).json({ ok: false, error: `Module load failed: ${modulePath}: ${e.message}` });
      }
    }
    router(req, res, next);
  };
}
```

`resolveArgs` 헬퍼도 파일 상단에 추가:

```javascript
// server/index.cjs — 신규 추가 (makeLazy 근처)
function resolveArgs(argNames, ctx) {
  return argNames.map(name => {
    if (name === 'db') return ctx.db;
    if (name === 'appDataPath') return ctx.appDataPath;
    if (name === 'BASE_DIR') return ctx.BASE_DIR;
    return name;
  });
}
```

`registerLazyApplication()` 내부는 registry 기반으로 간소화:

```javascript
function registerLazyApplication() {
  const { db, appDataPath } = require('./database.cjs');
  const ctx = { db, appDataPath, BASE_DIR };

  // Tier 0: 즉시 등록 — registry 순회
  for (const entry of routeRegistry.filter(r => r.tier === 0)) {
    const router = require(entry.module)(...resolveArgs(entry.args, ctx));
    app.use(entry.path, router);
  }

  if (IS_MINIMAL_BUILD) {
    console.log('[Server] minimal build mode: auth routes only');
    return postStartupCtx;
  }

  // BigQuery 감시 prefix: registry에서 자동 추출
  const BIGQUERY_IMMEDIATE_SYNC_PREFIXES = routeRegistry
    .filter(r => r.watch)
    .map(r => r.path);

  // BigQuery 감시 미들웨어 — 기존 로직 유지
  app.use((req, res, next) => { /* ... BIGQUERY_IMMEDIATE_SYNC_PREFIXES 사용 ... */ });

  // Tier 1: lazy wrapper 등록
  const tier1Entries = routeRegistry.filter(r => r.tier === 1);
  const tier1RouterRefs = {};
  for (const entry of tier1Entries) {
    tier1RouterRefs[entry.path] = null;
    app.use(entry.path, (req, res, next) => {
      if (!tier1RouterRefs[entry.path]) {
        try {
          tier1RouterRefs[entry.path] = require(entry.module)(...resolveArgs(entry.args, ctx));
        } catch (e) {
          console.error(`[Lazy] Tier1 로드 실패 ${entry.path}:`, e.message);
          tier1RouterRefs[entry.path] = 'error';
          return res.status(500).json({ ok: false, error: `Module load failed: ${entry.path}` });
        }
      }
      if (tier1RouterRefs[entry.path] === 'error') {
        return res.status(500).json({ ok: false, error: `Module previously failed: ${entry.path}` });
      }
      tier1RouterRefs[entry.path](req, res, next);
    });
  }

  // Tier 2: makeLazy 등록 — registry 순회 (기존 11줄 제거)
  for (const entry of routeRegistry.filter(r => r.tier === 2)) {
    app.use(entry.path, makeLazy(entry.module, ...resolveArgs(entry.args, ctx)));
  }

  // preload-trigger: registry 기반 자동화
  app.post('/api/preload-trigger', (req, res) => {
    res.json({ ok: true });
    setImmediate(() => {
      let i = 0;
      function loadNext() {
        if (i >= tier1Entries.length) { startBigQueryScheduler(); return; }
        const entry = tier1Entries[i++];
        if (!tier1RouterRefs[entry.path]) {
          try {
            tier1RouterRefs[entry.path] = require(entry.module)(...resolveArgs(entry.args, ctx));
          } catch (e) {
            console.error(`[Preload] Tier1 로드 실패 ${entry.path}:`, e.message);
            tier1RouterRefs[entry.path] = 'error';
          }
        }
        setTimeout(loadNext, 200);
      }
      loadNext();
    });
  });
}
```

### 5.4. 효과

| 항목 | 현재 | 개선 후 |
|------|------|---------|
| 새 라우트 추가 시 수정 위치 | index.cjs 내 2~3곳 | `routeRegistry.cjs` 배열에 **1줄 추가** |
| 계층 구조 파악 | index.cjs 흩어짐 | registry 배열로 즉시 파악 |
| BigQuery 감시 prefix | 별도 배열 유지보수 | registry의 `watch` 플래그로 자동 생성 |
| preload-trigger 대상 | tier1RouteConfigs 수동 관리 | registry의 tier 1 항목으로 자동 대응 |
| 모듈 로드 실패 시 | null 상태로 매 요청 재시도 | 'error' 상태 저장 → 500 응답 (무한 재시도 방지) |

### 5.5. 새 라우트 추가 절차 (개선 후)

#### ① 라우트 파일 생성

## 6. 📋 전체 수정 작업 계획 (수정된 우선순위)

| 우선순위 | 작업 | 설명 | 위험도 |
|---------|------|------|--------|
| **P0** | `.cjs` 파일 CP949 → UTF-8 변환 | PowerShell 1회 실행, 단순 변환 | ✅ 안전 |
| **P1** | `electron-builder.config.js` - `npmRebuild: true` 추가 | better-sqlite3 ABI 불일치 해결 (서버 사망 직결) | ✅ 1줄 추가 |
| **P2** | `uploadRoutes.cjs` 매개변수명 `baseDir` → `appDataPath` | Program Files EPERM 해결 | ✅ 2줄 변경 |
| **P3** | `routeRegistry.cjs` 생성 + `index.cjs` 리팩터 | 아키텍처 개선 (가장 큰 변경) | ⚠️ 중간 규모 |
| **P4** | 릴리즈 재빌드 및 테스트 | `npm run build && npm run electron:build` | ✅ |

### 실행 순서 근거

1. **P0 → P1**: better-sqlite3 fix가 인코딩 바로 다음인 이유
   - DB가 안 뜨면 서버 전체가 죽음 → 어떤 API도 동작 불가
   - 인코딩을 고쳐도 DB 로드 실패하면 소용없음

2. **P1 → P3**: rebuild 설정 후에 아키텍처 개선하는 이유
   - `routeRegistry.cjs` 리팩터 중에도 `npm run electron:build`로 중간 테스트 가능
   - better-sqlite3가 먼저 고정되어야 빌드 테스트가 의미 있음

3. **P3**: 아키텍처 개선은 가장 큰 변경이므로 **P0~P2가 안정화된 후** 진행

## 8. 🔍 서버 시작 시퀀스 (참고: routeRegistry 리팩터 완료 후 기준)

```
1. Electron main.cjs
   ├── fork('server.cjs', { env: { ELECTRON: '1', OSOO_SERVER_TOKEN } })
   └── waitForLocalServer (port file polling + /api/ping)

2. server.cjs → server/index.cjs
   ├── listen(port) → writePortFileEarly(port)
   └── setImmediate → registerLazyApplication()
       ├── require('./database.cjs')  ← ★ better-sqlite3 로드 (P1 필수)
       ├── routeRegistry 순회하며 Tier 0/1/2 등록 (P3 후 적용)
       └── schedulePostStartupTasks()

3. Renderer (React)
   ├── initServerConfig() → port 발견
   ├── GET /api/auth/me → Tier 0
   ├── POST /api/auth/local-login → Tier 0
   └── POST /api/preload-trigger → Tier 1 프리로드 (200ms 간격)
```

> **현재 상태와의 차이**: 현재(`index.cjs` 리팩터 전)는 Tier 0/1/2가 `index.cjs`에 직접 코딩되어 있음.  
> 리팩터 후에는 `routeRegistry.cjs` 배열을 순회하는 방식으로 변경됨.
