- 파샬플롬이 있는 현장과 없는 현장의 일일업무일지 유입량·공정유입량 바인딩을 각각 검증했다.
- 현장 위치 조회 PowerShell 리소스와 위치 저장 API를 실제 호출하여 검증했다.
- `npm run validate`, `npm run validate:api`, `npm run build`, `npm run electron:build`, `npm run validate:asar`를 통과했다.
- 표준 자동 업데이트 설치파일과 자격증명 포함 현장용 통합 설치파일 1.0.3을 생성했다.

## 작업 인계 및 최종 배포 계획

현재 대화 에이전트의 작업 가능 시간이 얼마 남지 않아, 이 문서에 정리된 코드 수정은 다른 개발 에이전트에게 인계한다.

### 다른 개발 에이전트의 역할

- 이 문서의 TODO를 기준으로 원인을 다시 확인한 뒤 코드를 수정한다.
- 항목을 임의로 생략하거나 화면 증상만 임시로 가리지 않는다.
- 로컬 DB를 설정의 단일 기준으로 삼고, 저장·재조회·매핑·업무 그리드가 같은 데이터를 사용하도록 연결한다.
- `AGENTS.md`, `LAYOUT_CONTRACT.md`, `POST_RELEASE_MAINTENANCE_GUIDE.md`를 준수한다.
- 기존 MVVM 구조와 설정 패널·위젯·hook·service 구조 안에서 작은 증분 수정으로 처리한다.
- 보호 대상 구조나 파일을 변경해야 한다면 사용자에게 먼저 승인을 받는다.
- 개발 PC의 기존 DB나 시험 데이터가 수정 결과를 우연히 정상처럼 보이게 하지 않도록 신규 DB 환경도 별도로 검증한다.

### 구현 우선순위

1. 신규 설치의 현장 미확정 초기 상태
2. 기본설정 위젯의 DB 저장·재조회 및 사용자 추가 항목 보존
3. 엑셀 원본 파싱, 특정 행 미리보기 및 시트 목록 자료형 통일
4. 유량·약품·키트·수질 매핑 저장과 재진입 복원
5. 실제 데이터 저장 건수 검증 및 업무 그리드 캐시 갱신
6. 동적 검침항목·분석장소의 매핑, 관리 그리드 및 통합 입력 모달 반영
7. QnTECH 중복 수질값, 사진 보강, 키트 사용량 및 과거 재고 연쇄 계산
8. 성적서의 확정 현장 자동 필터와 연·월 필터
9. 나머지 현장 배포 오류와 UI 문제

### 수정 에이전트의 완료 조건

- 각 TODO 항목에 수정 파일, 원인, 수정 내용 및 검증 결과를 기록한다.
- 임시 성공 메시지가 아니라 DB 저장 결과를 다시 조회하여 검증한다.
- 신규 설치 환경과 기존 DB 보존 환경을 모두 시험한다.
- `npm run validate`를 통과한다.
- UI 또는 레이아웃을 변경했다면 관련 화면을 실제 실행하여 확인한다.
- 최종 검토 전에는 버전 변경, 설치파일 생성, GitHub Release 교체 또는 배포를 수행하지 않는다.
- 수정 완료 후 사용자와 현재 대화 에이전트가 검토할 수 있도록 변경사항을 커밋하지 않은 상태 또는 명확한 별도 커밋으로 남긴다.

### 현재 대화 에이전트가 이후 수행할 역할

다른 에이전트의 코드 수정이 완료되면 현재 대화 에이전트가 다시 이어서 다음 작업을 수행한다.

1. 전체 변경 diff와 수정 파일을 검토한다.
2. TODO별 요구사항이 실제 코드에 반영됐는지 대조한다.
3. 저장·재조회·매핑·업무 그리드·통합 모달의 전체 워크플로우를 검증한다.
4. `npm run validate`와 `npm run build`를 실행한다.
5. 필요한 개발서버 및 Electron 실행 검증을 수행한다.
6. 신규 설치 환경을 기준으로 통합 설치패키지를 다시 만든다.
7. 설치파일 내부에 앱 본체, 기본 DB 시드, 일지 양식 및 필수 리소스가 포함됐는지 확인한다.
8. 버전과 자동 업데이트 파일을 점검한다.
9. 최종 승인 후 GitHub에 커밋·푸시하고 새 Release 자산을 업로드한다.
10. 현장 배포 방법 문서와 실제 설치파일명이 일치하도록 갱신한다.

주의:

- 현장 배포와 GitHub Release는 코드 수정이 끝났다는 이유만으로 바로 진행하지 않는다.
- 최종 검토에서 문제가 발견되면 수정 에이전트의 결과를 보완한 뒤 다시 전체 검증한다.
- 아직 현장에 정상 배포된 설치본이 없으므로, 다음 설치파일은 신규 설치 흐름을 최우선으로 검증해야 한다.

## 1. 최초 설치인데 기존 현장 변경 경고가 표시됨

상태: **수정 필요**

현상:

- admin으로 로그인해 설정 메뉴에 들어가면 현장 드롭다운에 이미 어떤 현장이 선택되어 있다.
- 천안휴게소를 선택하면 다음 취지의 경고가 표시된다.

```text
이미 현장이 저장된 상태입니다.
선택한 현장으로 변경하시겠습니까?
```

원인 후보 및 확인 결과:

- 신규 DB 생성 시 `app_settings.site_name`에 `새 현장`을 넣는 시드가 존재한다.
- 빈 `app_settings.site_id`에 임의 UUID를 자동으로 넣는 마이그레이션이 존재한다.
- UI는 `site_id` 또는 `site_name`이 있으면 앱에 현장이 이미 설정된 것으로 판단한다.
- 개발 PC에는 이전 테스트 현장인 `횡성휴게소(인천방향)` 설정도 남아 있다.

원하는 동작:

- 최초 설치에서는 현장이 선택되지 않은 빈 상태여야 한다.
- 드롭다운은 `현장을 선택하세요` 상태로 시작해야 한다.
- admin의 최초 현장 선택에는 변경 경고를 표시하지 않는다.
- 현장 확정 및 현장관리자 로컬 저장이 완료된 이후, 다른 현장으로 변경할 때만 경고한다.
- 기존 운영 PC의 실제 현장 설정은 업데이트 시 보존해야 한다.

수정 시 검토 대상:

- `server/database.cjs`의 `app_settings` 초기 시드
- 빈 `site_id`에 UUID를 자동 입력하는 마이그레이션
- 설정 화면의 `hasLoadedSettings`, `isAppSiteConfigured` 판정
- 신규 설치와 기존 운영 DB를 구분하는 방법

## 2. 현장 선택 후 현장관리자 정보 로드

상태: **정상 확인**

확인 내용:

- 드롭다운에서 현장을 선택한 뒤 해당 현장관리자 정보를 불러오는 동작은 정상적으로 수행됐다.

유지할 동작:

- Google Sheets의 현장 및 회원 정보를 기준으로 선택 현장의 관리자를 조회한다.
- 최종 현장 확정 시 해당 현장관리자의 로그인 정보를 로컬 DB에 저장한다.
- 이후 현장관리자는 인터넷 연결 여부와 관계없이 로컬 인증을 사용한다.
- admin 계정은 로컬에 저장하지 않고 항상 Google Sheets에서 인증한다.

## 3. 파샬플롬 유량계가 있는 현장의 유량 구분 및 일일업무일지 바인딩

상태: **미구현 — 구현 필요**

업무 규칙:

- 일부 현장에는 기본 유입유량계보다 앞단에 `파샬플롬 유량계`가 추가로 설치되어 있다.
- 파샬플롬 유량계가 있는 현장에서는 다음과 같이 구분한다.

| 검침 항목 | 업무상 의미 | 일일업무일지 바인딩 |
|---|---|---|
| 파샬플롬 유량계 | 현장으로 들어오는 전체 유입량 | 전날 유입, 금일 유입, 유입량 |
| 유입유량계 | 처리 공정으로 실제 유입되는 공정유입량 | 전날 공정, 금일 공정, 공정량 |

- 파샬플롬 유량계가 없는 현장은 기존처럼 유입유량계를 일반 유입량으로 사용한다.
- 즉, 파샬플롬 유량계가 없는 현장에서는 `유입유량계` 검침값을 전날 유입, 금일 유입, 유입량에 바인딩하고 공정유입량 칸은 별도로 만들지 않는다.
- 한글 일지 양식에는 공정유입량용 `전날 공정`, `금일공정`, `공정량` 책갈피가 이미 마련되어 있다.

구현 시 확인할 사항:

- 현장 설정에 파샬플롬 유량계 사용 여부와 검침 항목을 영구 저장한다.
- 통합 입력 모달과 유량관리 화면에서 두 유량계를 별도 항목으로 표시한다.
- 전날 검침값, 금일 검침값 및 일일 사용량 계산을 각각 독립적으로 처리한다.
- 일일업무일지 생성 시 현장 설정에 따라 유입량과 공정유입량 책갈피를 올바르게 바인딩한다.
- 기존 현장과 기존 데이터의 유입유량 의미가 변경되지 않도록 호환 규칙을 둔다.
- BigQuery 동기화 필드와 중앙관리자 앱의 조회·분석 필드도 두 유량을 구분할 수 있는지 확인한다.

## 4. 검침항목 관리 위젯에서 추가 항목이 저장 후 사라짐

상태: **수정 필요**

재현 절차:

1. 현장 기본 설정의 검침항목 관리 위젯에서 `파샬플롬 유량계`를 추가한다.
2. 추가된 항목은 기본적으로 체크된 상태로 표시된다.
3. 설정을 저장한다.
4. 저장 후 검침항목 관리 위젯에서 해당 항목이 사라진다.
5. 유량설정 탭에도 해당 항목이 나타나지 않아 엑셀 열 매핑을 할 수 없다.

원하는 동작:

- 추가한 검침항목의 이름, 활성화 여부, 순서 및 유형이 저장되어야 한다.
- 저장 직후 위젯을 다시 조회해도 파샬플롬 유량계가 유지되어야 한다.
- 활성화된 항목은 유량설정 탭의 매핑 대상에 즉시 나타나야 한다.
- 앱 재시작 후에도 항목과 매핑이 유지되어야 한다.
- 체크 해제는 항목 삭제가 아니라 비활성화로 처리하고 기존 데이터와 매핑은 보존해야 한다.

원인 확인 대상:

- 위젯의 임시 상태와 저장 요청 데이터 사이에서 사용자 추가 항목이 누락되는지 확인
- 저장 API가 사전에 정의된 기본 검침항목만 허용하거나 필터링하는지 확인
- 저장 후 재조회 과정에서 사용자 정의 항목을 기본 항목 목록으로 덮어쓰는지 확인
- 유량설정 탭이 고정 상수 목록만 사용하고 있는지 확인
- 로컬 DB 저장 구조와 엑셀 매핑 설정이 사용자 정의 검침항목을 지원하는지 확인

코드 확인 결과:

- 사용자 추가 검침항목은 추가 즉시 로컬 DB의 `config_items`에 저장된다.
- 그러나 설정을 다시 불러올 때 저장된 검침항목 목록과 코드의 기본 검침항목 목록이 정확히 같지 않으면 `needsResyncFlowItemsForSite()`가 실행된다.
- 이때 사용자 정의 항목을 포함한 저장 목록 대신 기본 검침항목만 다시 만들고 유량 매핑도 초기화한다.
- 따라서 `파샬플롬 유량계`처럼 정상적으로 추가·저장된 사용자 정의 항목도 재조회 시 사라질 수 있다.
- 첫 번째 저장 직후에는 DB에 파샬플롬 항목이 남아 있더라도 화면 상태만 기본 목록으로 초기화될 수 있다.
- 그 상태에서 기본설정 저장 버튼을 다시 누르면 화면의 기본 목록만 서버로 전달되고, 서버가 전달 목록에 없는 파샬플롬 항목을 DB에서 삭제한다.
- 즉 `DB 저장 → 재조회 중 UI 누락 → 다음 저장에서 DB 실제 삭제`의 2단계 데이터 손실 가능성이 있다.
- 기본항목 동기화는 누락된 기본항목만 보충해야 하며 사용자 정의 항목과 기존 매핑을 제거하면 안 된다.
- 저장 후에는 서버가 반환하거나 재조회한 DB 목록을 그대로 화면에 복원하고, DB와 UI 항목 목록이 다르면 성공 메시지를 표시하지 않도록 검증한다.

확정된 수정 원칙:

- 검침항목, 약품항목, 키트항목 및 분석장소는 로컬 DB를 단일 기준으로 삼는다.
- 코드에 정의된 기본항목은 신규 DB를 처음 생성할 때만 시드한다.
- 설정 화면은 매번 DB의 `config_items`를 조회해 그대로 표시한다.
- 사용자가 위젯에서 추가·수정·순서 변경·활성화 변경한 결과는 DB에 저장하고, 저장 후에도 DB 값을 다시 표시한다.
- 앱 실행이나 설정 재조회 때 코드의 기본 목록으로 DB 목록을 덮어쓰지 않는다.
- 업데이트로 새로운 필수 기본항목을 추가해야 할 때만 기존 사용자 항목을 보존하면서 누락된 항목을 보충하는 명시적 마이그레이션을 사용한다.
- 이미 업무 데이터나 엑셀 매핑이 연결된 항목은 물리적으로 삭제하기보다 비활성화하여 이력과 매핑을 보존한다.
- 같은 원칙을 검침항목뿐 아니라 약품, 키트 및 분석장소 위젯에도 적용한다.
- 배포본에 개발 PC의 실제 `osoo.db` 파일을 포함하지 않는다.
- 설치 후 신규 DB가 생성될 때 각 위젯의 기본항목을 `config_items`에 시드하고, 이후에는 해당 DB 값만 사용한다.
- 이렇게 하면 개발 현장의 선택값, 관리자 정보, 시험 데이터 및 기존 매핑이 신규 현장에 함께 배포되는 것을 방지할 수 있다.

## 5. 사용자 추가 분석장소가 수질 매핑에 나타나지 않거나 유지되지 않음

상태: **확인 및 수정 필요**

현장 사례:

- 해당 현장에서는 `무산소조`와 `혐기조`를 모두 실험한다.
- 기본 분석장소에 없는 `혐기조`를 사용자가 추가했다.
- 추가 후 수질설정 탭에서 혐기조에 해당하는 엑셀 열을 매핑할 수 없었던 것으로 보인다.

원하는 동작:

- 사용자가 추가한 혐기조가 분석장소 목록에 저장되어야 한다.
- 기본설정 저장 및 앱 재시작 후에도 혐기조와 활성화 상태 및 순서가 유지되어야 한다.
- 수질설정 탭에서 활성 분석장소를 기준으로 암모니아성질소, 질산성질소, 알칼리도 등의 매핑 행을 생성해야 한다.
- 수질 데이터 저장 및 QnTECH 데이터 연결에서도 같은 분석장소 이름을 사용해야 한다.
- 현장별로 실험하지 않는 장소는 비활성화할 수 있어야 하며 기존 데이터는 삭제하지 않는다.

코드 확인 결과:

- 분석장소도 로컬 DB의 `config_items` 테이블에 `category = 'location'`으로 저장한다.
- 분석장소 저장 로직은 사용자 정의 장소를 보존하도록 작성되어 있으며, 수질 매핑 화면도 활성화된 장소를 동적으로 사용한다.
- 따라서 혐기조가 실제로 사라졌다면 현장 선택 시 기본값 재설정, 저장 후 재조회, 활성 상태 또는 화면 상태 갱신 과정 중 어느 단계에서 누락되는지 재현 검증이 필요하다.
- 인산염인(PO4-P)은 현재 공법별 고정 장소만 매핑하도록 별도 제한되어 있으므로, 혐기조를 포함한 사용자 정의 장소에 인산염인도 매핑해야 하는지는 업무 규칙을 별도로 확정해야 한다.

## 6. 기본설정 저장 버튼의 저장 범위 확정

상태: **저장 범위 확정 — 구현 검증 필요**

확정된 저장 원칙:

- 각 설정 탭은 해당 탭의 저장 버튼으로 자기 탭의 설정만 저장한다.
- 기본설정 탭의 저장 버튼은 다른 탭의 엑셀 매핑이나 데이터 불러오기를 저장하지 않는다.
- 기본설정 탭의 저장 버튼 한 번으로 기본설정 탭 안에서 입력·변경한 다음 항목을 저장한다.
  - 선택 현장과 현장관리자
  - 공법 및 계열
  - 검침항목과 활성화 상태 및 순서
  - 약품항목과 활성화 상태 및 순서
  - 키트항목과 활성화 상태
  - 분석장소와 활성화 상태 및 순서
  - 약품 기본량
  - 키트 기본량
  - 기본설정 탭에 속하는 기타 위젯 설정값
- 유량·약품·키트·수질 등의 엑셀 열 매핑은 각 매핑 탭의 저장 버튼이 담당한다.

현재 코드 동작:

- 기본설정 저장 버튼은 `/api/settings`를 호출한다.
- 이 요청에는 현장정보와 `flow`, `medicine`, `location` 항목만 포함된다.
- `kit` 항목은 기본설정 저장 요청에 포함되지 않는다.
- 약품 기본량은 `입고량지정` 모달의 별도 저장 버튼으로 `/api/settings/medicine-defaults`에 저장한다.
- 키트 기본량은 `키트량지정` 모달의 별도 저장 버튼으로 `/api/settings/kit-defaults`에 저장한다.
- 유량·약품·키트·수질 엑셀 매핑은 각 설정 탭의 별도 저장 버튼/API로 저장하며, 이 분리는 정상적인 설계이다.
- 문제는 다른 탭의 매핑을 저장하지 않는 것이 아니라, 기본설정 탭 내부 값 중 일부가 기본설정 저장 범위에서 빠져 있거나 별도 저장 상태와 일관되지 않은 점이다.

저장 위치:

- 운영 설정의 주 저장소는 `%APPDATA%\Osoo_Handle_App\osoo.db` 로컬 SQLite DB이다.
- 현장 기본정보와 엑셀 범위 설정은 `app_settings` 테이블에 저장한다.
- 검침항목, 약품, 키트, 분석장소, 활성화 상태, 표시 순서, 매핑 열 및 기본량은 `config_items` 테이블에 저장한다.
- `config` 폴더의 `.env.local`과 JSON 파일들은 서비스 자격증명 및 실행환경 설정이며, 현장 위젯 설정을 저장하는 파일이 아니다.

수정 방향:

- 각 탭의 저장 책임은 현재처럼 분리한다.
- 기본설정 탭에서 변경 가능한 값은 기본설정 저장 시 함께 저장하여 일부만 저장되는 상태를 막는다.
- 기본설정 탭 내부의 약품·키트 기본량 모달은 즉시 저장 방식과 최종 기본설정 저장 방식 중 하나로 규칙을 통일한다.
- 저장 후 서버 응답을 다시 조회해 검침항목, 약품, 키트, 분석장소, 기본량 및 순서가 모두 동일한지 검증한다.

## 7. 엑셀의 실제 시작행과 매핑 미리보기 행이 일치하지 않음

상태: **현장 재현 확인 — 원인 분석 및 수정 필요**

현상:

- 기본설정 탭에서 기존 엑셀 원본은 정상적으로 받아들였다.
- 2026년 1월 1일부터 가져오기 위해 시작행을 약 369, 종료행을 약 542로 지정했다.
- 해당 행이 엑셀에 실제로 존재하는데도 매핑 화면의 시작행 미리보기에 `No Data`가 표시됐다.
- 우연히 시작행을 200으로 바꾸자 해당 행의 데이터는 미리보기에 표시됐다.
- 유량, 약품, 키트 등의 여러 탭에서 같은 현상이 있었다.
- 따라서 특정 행의 미리보기 실패는 사용자 입력 착오가 아니라 현장에서 실제 확인된 문제로 취급한다.

현재 코드 확인:

- 원본 엑셀은 AppData에 복사된 뒤 셀 데이터를 `excel_raw_data` 테이블에 원래 엑셀 행 번호와 함께 저장한다.
- 미리보기는 선택 시트명과 시작행 번호가 정확히 일치하는 `excel_raw_data` 행만 조회한다.
- 파서는 원본 워크북의 앞쪽 3개 시트만 저장한다.
- 각 시트는 A열을 날짜 열로 가정하고, 3행부터 어제 날짜까지의 범위만 저장한다.
- 업로드 직후 서버는 시트 목록을 문자열 배열로 반환하지만, 이후 `excel-status` 재조회는 `{ sheet_name, max_row, imported_at }` 객체 배열을 반환한다.
- 화면의 시트 선택 컴포넌트는 문자열 배열만 처리하므로 재조회 후 객체가 들어오면 시트명 값이 깨지거나 기존 선택값과 일치하지 않을 수 있다.
- 따라서 다음 경우 미리보기가 비어 있을 수 있다.
  - 선택한 시트가 앞쪽 3개 시트에 포함되지 않은 경우
  - 해당 시트의 날짜가 A열이 아닌 경우
  - A열 날짜를 ExcelJS가 날짜로 해석하지 못한 경우
  - 파서가 계산한 `max_row`보다 큰 행을 지정한 경우
  - 화면에서 선택한 시트명과 DB에 저장된 시트명이 다른 경우

수정 방향:

- 앞쪽 3개 시트 제한을 제거하거나 실제 선택 대상 시트를 모두 저장한다.
- 시트별 날짜 열을 고정 A열로 가정하지 않고 설정값 또는 탐지 결과를 사용한다.
- 업로드 응답과 상태 재조회 응답의 시트 목록 형식을 하나로 통일하고, 화면에서는 `sheet_name`을 명시적으로 사용한다.
- 매핑 화면에 DB에 저장된 해당 시트의 최대 행과 선택 행의 실제 날짜를 표시한다.
- 선택한 행이 저장 범위 밖이면 `No Data` 대신 구체적인 원인을 안내한다.
- 시작행 값이 없는 상태에서는 저장을 허용하지 말고, 값이 있는 가장 가까운 행을 안내한다.
- 현장에서 사용한 천안휴게소 원본 엑셀로 200행과 369행을 직접 비교 검증한다.

## 8. 약품·키트 매핑이 실제 값 없이 날짜 행만 성공 건수로 계산됨

상태: **원인 확인 — 수정 필요**

확인된 원인:

- 약품·키트 가져오기 로직은 시작행부터 종료행까지 날짜가 있으면 `rowResults = { date }`를 만든다.
- 매핑된 약품·키트 값이 하나도 없더라도 이 날짜 전용 행을 `importedData`에 추가한다.
- 저장 완료 건수는 실제 DB에 값이 저장된 건수가 아니라 `importedData.length`를 사용한다.
- 따라서 실제 약품·키트 값이 0건이어도 `백몇십 개 저장 완료`처럼 표시될 수 있다.
- `저장된 데이터 보기` 모달에는 날짜 외 필드가 없으므로 사진처럼 `날짜` 열만 나타난다.

추가 원인 후보:

- 매핑 키의 항목명이 기본설정의 활성 약품·키트 이름과 정확히 일치하지 않으면 저장 전에 매핑 전체 또는 일부가 필터링된다.
- 기본설정 위젯의 항목 초기화 문제와 결합하면 화면에서는 매핑했지만 서버가 허용 항목으로 인정하지 않을 수 있다.

수정 방향:

- 값이 하나 이상 실제 저장된 행만 성공 목록에 포함한다.
- 처리 행 수, 날짜 확인 행 수, 실제 저장 행 수, 실제 저장 셀/항목 수를 구분해 표시한다.
- 활성 항목 불일치로 제외된 매핑 이름을 오류 또는 경고로 보여준다.
- 실제 저장값이 0건이면 성공으로 처리하지 않고 매핑과 시작행을 다시 확인하도록 안내한다.

## 9. 엑셀 가져오기 후 업무 메뉴가 기존 빈 캐시를 계속 표시함

상태: **원인 확인 — 수정 필요**

현상:

- 매핑 저장 완료 후 유량관리, 약품관리, 키트관리 메뉴로 이동해도 그리드는 초기 빈 상태였다.
- admin 계정이라 데이터가 보이지 않는 것으로 의심했으나, 현재 조회 API는 admin 여부로 데이터를 숨기지 않는다.

확인된 원인:

- 유량, 약품, 키트 Model은 조회 결과를 메모리 캐시에 보관한다.
- 설정 화면의 엑셀 가져오기는 DB 테이블에 직접 데이터를 저장하지만 각 업무 Model의 조회 캐시를 비우지 않는다.
- 가져오기 전에 해당 업무 메뉴가 빈 데이터를 한 번 조회했다면, 가져오기 후에도 그 빈 캐시를 계속 반환할 수 있다.

수정 방향:

- 각 매핑 가져오기 성공 시 해당 업무 Model 캐시를 무효화한다.
- 또는 업무 메뉴 진입 시 현장 확정 직후와 초기 데이터 가져오기 직후에는 강제 재조회한다.
- 서버 저장 결과와 업무 메뉴 조회 결과의 건수 및 날짜 범위를 비교 검증한다.
- admin과 현장관리자 모두 동일 로컬 DB를 보되, 현장 범위는 확정된 `app_settings.site_id`를 기준으로 일관되게 적용한다.

## 10. 저장 데이터 보기 모달이 날짜만 표시되고 화면이 찌그러짐

상태: **원인 확인 — 수정 필요**

확인된 원인:

- 모달 열 이름은 첫 번째 데이터 객체의 키만 사용한다.
- 첫 번째 객체가 `{ date: ... }`뿐이면 날짜 열만 생성되고, 뒤 행에 다른 값이 있어도 열을 표시하지 못한다.
- 가져오기 결과가 날짜 전용 행들로 구성되면 내용이 없는 좁은 표처럼 보인다.
- 모달 폭이 고정 600px이고, 많은 항목에 대한 가로 스크롤과 최소 너비 처리가 없다.

수정 방향:

- 전체 결과 행의 키를 합쳐 열 목록을 만든다.
- 날짜 외 값이 없는 행은 결과에서 제외하거나 `저장된 값 없음`으로 안내한다.
- 항목이 많으면 내부 가로 스크롤과 표 최소 너비를 적용한다.
- 실제 DB 저장 결과를 다시 조회하여 보여주고, 일시적인 프론트 상태만 표시하지 않는다.

## 11. 탭을 떠났다가 돌아오면 저장한 매핑이 초기화됨

상태: **확인 및 수정 필요**

관련 코드 흐름:

- 매핑 저장 성공 직후 전체 설정을 다시 조회한다.
- 유량 항목 목록이 코드 기본 목록과 다르면 사용자 항목과 유량 매핑을 초기화하는 문제가 이미 확인됐다.
- 약품·키트 매핑은 DB의 `config_items.excel_cell`에서 복원하지만, 기본 위젯 항목과 매핑 항목 이름이 불일치하면 화면 복원이 누락될 수 있다.
- 설정 재조회와 탭 전환이 동시에 일어나면서 빈 상태가 먼저 표시될 가능성도 있다.
- 엑셀 업로드 직후 시트 목록은 문자열이지만 탭 재진입 후 상태 조회 결과는 객체이므로, 시트 선택 컴포넌트가 기존 시트명을 복원하지 못하는 명확한 자료형 불일치가 있다.

원하는 동작:

- 각 탭 저장 후 해당 탭의 시트, 시작행, 종료행, 날짜 열 및 모든 항목 매핑을 DB에서 다시 읽어 동일하게 표시한다.
- 다른 탭으로 이동하거나 앱을 재시작해도 저장된 매핑이 유지되어야 한다.
- 저장 직후 DB 복원 결과가 화면의 저장 직전 값과 다르면 성공 메시지를 표시하지 않는다.

## 12. QnTECH 수질·사진 가져오기와 키트 재고 연쇄 계산 재검증

상태: **핵심 동작 구현 확인 — 현장 범위 보강 및 통합 검증 필요**

확정된 업무 규칙:

- 통합 입력 모달에서 날짜를 지정하고 QnTECH 수질 데이터와 측정사진을 가져온다.
- 같은 현장·같은 날짜에 같은 분석장소, 분석항목 및 분석값이 이미 저장되어 있으면:
  - 기존 수질값은 유지하거나 동일 값으로 덮어써도 된다.
  - 측정사진은 내려받아 로컬과 Drive에 보강한다.
  - 키트 사용량은 다시 증가시키지 않는다.
  - 키트 재고도 다시 차감하지 않는다.
- 같은 날짜에 일치하는 수질 데이터가 없으면:
  - 수질 데이터와 측정사진을 저장한다.
  - 실제 분석값이 존재하는 항목별로 키트 사용량을 반영한다.
  - 키트 재고를 계산한다.
- 과거 날짜의 새 수질 데이터를 가져와 키트 사용량이 추가되면:
  - 해당 날짜의 재고만 바꾸는 것이 아니라 그 날짜 이후 오늘까지 존재하는 모든 키트 재고를 순서대로 다시 계산한다.
- 수질분석 없이 키트만 사용한 수동 사용량은 보존해야 한다.

현재 코드에서 확인된 정상 동작:

- QnTECH 값은 같은 현장과 날짜의 기존 수질행을 분석장소·항목코드·값으로 비교한다.
- 가져온 모든 수질값이 기존 값과 일치하면 `matchedExistingData`로 판정한다.
- 이 경우 수질값 저장을 건너뛰고 사진 다운로드와 Drive 업로드는 그대로 수행한다.
- 단일 날짜 가져오기 API는 `matchedExistingData`일 때 키트 사용량 동기화를 건너뛴다.
- 기간 가져오기도 완전히 일치한 날짜는 키트 동기화 대상에서 제외한다.
- 새 수질값을 저장한 경우 분석값이 있는 항목 수를 기준으로 키트 사용량을 계산한다.
- 기존 수동 키트 사용량보다 분석 기반 사용량이 작으면 기존 사용량을 유지하고, 분석 기반 사용량이 더 크면 필요한 수량까지 올린다.
- 키트 사용량이 변경되면 가져온 과거 날짜를 시작점으로 이후 `kit_logs` 행을 날짜순으로 순회하며 재고를 연쇄 재계산한다.

보강이 필요한 부분:

- 이 앱은 설정에서 확정된 단일 현장 전용이며 로컬 DB에 다른 현장의 운영 데이터가 들어오지 않는 구조이므로, 로컬 키트 재고 연쇄 계산은 키트명과 날짜 기준으로 처리해도 된다.
- 다른 현장의 데이터는 BigQuery를 사용하는 중앙관리자 앱에서만 조회한다.
- 단, 로컬 DB가 단일 현장 전용이라는 것과 동기화 데이터의 현장 식별은 별개이다.
- BigQuery로 전송하는 모든 업무 테이블의 각 행에는 확정 현장의 `site_id`가 반드시 저장되어야 한다.
- `site_name`은 표시와 확인용으로 함께 저장하되, 현장 구분과 조인 및 중복 판정의 기준 키는 `site_id`를 사용한다.
- 엑셀 초기 이관, 수동 입력, QnTECH 자동 가져오기, 사진 메타데이터 생성 및 BigQuery 재전송 모두 동일한 `site_id`를 사용해야 한다.
- `site_id`가 비어 있는 행은 BigQuery 동기화 전에 오류로 분류하고 전송하지 않거나 확정 현장 ID로 안전하게 보정해야 한다.
- 동일 데이터 판정은 기존 데이터에 추가 행이 있어도 가져온 행만 모두 찾으면 일치로 처리하므로, 같은 분석 회차 전체가 정말 동일한지 비교 범위를 더 명확히 해야 한다.
- 같은 날짜의 일부 값만 기존과 같고 일부 값이 새 값인 경우, 새 값 저장과 키트 사용량 계산 결과가 중복 없이 정확한지 테스트해야 한다.
- 사진을 반복 가져올 때 로컬 파일은 같은 이름으로 갱신되고 Drive 파일도 같은 이름이면 갱신되므로, 중복 파일이 늘어나지 않는지 실제 Drive에서 검증한다.

필수 검증 시나리오:

1. 같은 날짜·같은 수질값 재가져오기 → 사진만 보강, 키트 사용량과 재고 불변
2. 같은 날짜·일부 수질값 변경 → 수질값 갱신, 필요한 키트 사용량만 반영
3. 수질값이 없는 날짜 최초 가져오기 → 수질·사진·키트 사용량 저장
4. 과거 날짜 신규 가져오기 → 해당 날짜 이후 오늘까지 키트 재고 연쇄 재계산
5. 같은 날짜에 수동 키트 사용량이 더 많은 경우 → 수동 사용량 보존
6. 초기 엑셀 이관 데이터와 현장관리자의 이후 입력이 이어지는 경우 → 과거 날짜부터 현재 재고까지 연속 계산

## 13. 성적서 목록의 현장 선택 드롭다운 제거 및 확정 현장 자동 적용

상태: **수정 필요**

현상:

- 성적서 메뉴 우측 상단에 현장을 선택하는 드롭다운 컴포넌트가 표시된다.
- admin으로 접속하면 여러 현장을 선택하여 성적서 목록을 바꿀 수 있는 형태로 동작한다.

확정된 업무 규칙:

- 현장관리자 앱은 한 PC에 한 현장만 사용하는 현장 전용 앱이다.
- 현재 현장은 설정 메뉴에서 admin이 최초 확정한다.
- 성적서 메뉴는 별도의 현장 선택을 제공하지 않는다.
- 설정에서 확정된 현장의 성적서만 자동으로 조회하고 표시한다.
- 성적서 목록의 기간 필터는 화면 하단의 연도·월 선택기만 사용한다.
- 하단 선택기의 연도와 월을 변경하면 확정 현장의 해당 월 성적서만 다시 조회한다.
- 다른 현장의 성적서 조회는 중앙관리자 앱에서 담당한다.

수정 방향:

- 우측 상단 현장 드롭다운을 제거하거나 숨긴다.
- 성적서 조회 요청의 현장명 또는 현장 ID는 `app_settings`의 확정 현장을 자동 사용한다.
- admin 계정으로 로그인해도 확정 현장 외 성적서를 선택하거나 조회하지 못하도록 현장관리자 앱의 책임을 고정한다.
- 앱 초기 상태로 아직 현장이 확정되지 않았다면 성적서 목록 대신 `설정에서 현장을 먼저 확정해 주세요`를 표시한다.
- 연도·월 선택값과 조회 결과가 일치하는지 검증한다.

## 공통 데이터 식별 원칙

- 현장관리자 앱의 로컬 운영 범위는 확정된 한 현장이다.
- BigQuery 동기화 대상 데이터에는 반드시 `site_id`와 `site_name`을 기록한다.
- `site_id`는 현장의 고유 식별자이며 변경 가능한 표시명인 `site_name`보다 우선한다.
- 모든 신규 저장·업데이트·엑셀 이관·외부 서비스 가져오기에서 `app_settings.site_id`를 메타데이터로 주입한다.
- BigQuery upsert 키에는 날짜와 업무별 고유키뿐 아니라 `site_id`가 포함되어야 한다.

## 14. 사용자 추가 검침항목·분석장소의 업무 그리드 동적 반영

상태: **관리 그리드 연결 확인 — 통합 입력 모달 수정 필요**

확정된 요구사항:

- 기본설정의 검침항목 위젯에서 파샬플롬 유량계 등을 추가하면 유량관리 그리드에 해당 항목의 열이 자동으로 추가되어야 한다.
- 기본설정의 분석장소 위젯에서 혐기조 등을 추가하면 수질관리 그리드에서 각 수질항목 아래 혐기조 열이 자동으로 추가되어야 한다.
- 비활성화한 항목과 장소는 신규 입력용 그리드에서 숨기되 기존 저장 데이터는 삭제하지 않는다.
- 순서를 변경하면 업무 그리드와 통합 입력 모달에도 같은 순서로 반영한다.

코드 확인 결과:

- 유량관리 화면은 설정에서 불러온 활성 `flowItems`를 기준으로 그리드 열과 통합 입력 컨텍스트를 동적으로 생성한다.
- 따라서 DB 저장·재조회 초기화 문제만 해결되면 파샬플롬 유량계도 유량관리 그리드 열로 표시될 구조이다.
- 수질관리 화면도 설정에서 불러온 활성 `locationItems`를 기준으로 각 수질항목의 하위 열을 동적으로 생성한다.
- 따라서 혐기조가 DB와 설정 상태에 유지되면 수질관리 조회 그리드에는 혐기조 열이 표시될 구조이다.

확인된 누락:

- 통합 데이터 입력 모달의 수질 표는 분석장소를 다음 5개로 고정하고 있다.

```text
유량조정조, 무산소조, 포기조, 침전조, 방류조
```

- 이 때문에 혐기조처럼 사용자가 추가한 분석장소가 수질관리 조회 그리드에는 나타나더라도 통합 입력 모달에는 나타나지 않아 입력할 수 없다.

수정 방향:

- 통합 입력 모달도 전달받은 활성 분석장소 목록을 기준으로 열을 동적으로 생성한다.
- 고정 5열 레이아웃을 제거하고 장소 수에 따라 내부 가로 스크롤과 최소 열 너비를 적용한다.
- 유량관리 그리드, 수질관리 그리드, 통합 입력 모달 및 엑셀 매핑 탭이 모두 동일한 DB 항목 목록과 순서를 사용하게 한다.
- 파샬플롬 유량계와 혐기조를 추가한 뒤 저장, 재진입, 매핑, 입력, 조회까지 전체 흐름을 검증한다.
