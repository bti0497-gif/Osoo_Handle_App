## 2. 🔴 수정 필요 — 자동업그레이드 정책 (사용자 핵심 요구)

### 사용자 요구사항
> "로그인 안 된 상태(트레이)에서도 자동 업그레이드 후, 사용자가 로그인하면 '업그레이드를 실행합니다' 안내 후 앱 종료하고 새 버전 설치"
> 배경: 양방향 현장 1명 관리, 오전 9시엔 동시 로그인 불가 → 로그인 전 자동 설치 필수

### 현재 1.0.10 상태
- 매일 09:00 점검 + 자동 다운로드 ✅
- **설치는 오직 사용자가 StatusBar 수동 버튼 클릭할 때만 실행** (`installUpdate()` IPC 호출 필요).
- 즉 **로그아웃/트레이 상태에선 다운로드만 되고 설치는 안 됨.**

### 현장 사례 (7/5)
- 밤새 앱 2개 켜져 있었으나 로그아웃 상태 → 자동 설치 안 됨
- 사용자가 원격 접속해 한 개를 닫고, 나머지에 로그인한 뒤 **수동으로 업데이트 버튼 클릭** → 그제야 1.0.10 설치
- 자동업데이트가 아니라 **사용자 수동 설치**로 진행됨

### 해결 방안 (2단계)

**1단계: 트레이/로그아웃 상태에서 자동 설치 허용**
- `electron/updater.cjs`의 `update-downloaded` 이벤트 핸들러에서, 다운로드 완료 후 **idle 5분(또는 일정 대기 시간) 도달 시 자동 `quitAndInstall(false, true)` 실행**.
- 단, 설치 직전 `stopServerGracefully()`로 서버 자식 프로세스 안전 종료 (1.0.10에 이미 구현됨, `main.cjs:324-326`).
- 이게 1.0.9의 idle 자동설치 로직을 "안전하게" 되살리는 것. 단, 1.0.9의 idle 측정 결함(`before-input-event` 의존)은 `powerMonitor.getSystemIdleTime()` 단일 기준으로 단순화.

**2단계: 사용자 로그인 시 pending 업데이트 안내**
- 앱 시작 또는 로그인 성공 시 `hasDownloadedUpdate()` 체크.
- true면 모달/배너로 "업데이트가 준비됐습니다. 설치하시겠습니까?" 표시.
- 사용자 확인 시 `installUpdate()` 호출 → 안내와 함께 설치 진행.
- 이 흐름이 사용자가 원하는 "로그인하면 업그레이드 안내 후 설치"에 부합.

**예상 결과**:
- 밤새 켜져 있으면 idle 5분 도달 시 자동 설치 (1단계)
- 아침에 사용자가 먼저 앱을 만지면 자동 설치가 미뤄지더라도, 로그인 직후 안내 모달로 설치 유도 (2단계)
- 두 단계가 합쳐져 어느 시나리오든 업데이트가 보장됨

## 3. 🔴 수정 필요 — 트레이 앱 중복 실행 (single instance lock 누데이트)

### 현상
- 7/5 현장에서 트레이에 앱 아이콘이 2개 뜸.
- 사용자가 한 개를 수동으로 닫고 나머지를 사용.

### 원인
- `electron/main.cjs`에 **`requestSingleInstanceLock()` 코드가 없음.**
- 두 번 실행되면 두 프로세스가 같은 DB/포트를 잡고 충돌 위험 (데이터 손상, DB 잠금).

### 해결 방안
`app.whenReady()` 호출 **이전**에 추가:
```js
const gotTheLock = app.requestSingleInstanceLock();
if (!gotTheLock) {
  app.quit();
  return; // 또는 process.exit(0)
}
app.on('second-instance', () => {
  // 이미 실행 중인 창을 포커스
  if (mainWindow) {
    if (mainWindow.isMinimized()) mainWindow.restore();
    mainWindow.show();
    mainWindow.focus();
  }
});
```

### 주의
- `AGENTS.md`에 `electron/` 내 파일 수정은 **사용자 확인 필수** 항목. 진행 전 사용자 승인 필요.
- 단일 인스턴스 락은 Electron 표준 패턴이라 구조 변경 아님.

## 4. 🟡 수정 필요 — 진단 로그 app_version 버그

### 현상
- 7/2~7/5 모든 진단 로그의 `app_version` 필드가 `?`(빈 문자열).
- 1.0.9와 1.0.10 구분이 로그만으로 불가능.

### 원인
`server/services/diagnosticLogService.cjs:43-50`:
```js
function getAppVersion() {
  try {
    const pkg = require('../../package.json');
    return pkg.version || '';
  } catch (_) {
    return '';  // ← asar 패키징 시 여기로 빠짐
  }
}
```

- 서버의 `cwd`는 `app.asar.unpacked` (`main.cjs:97`).
- `require('../../package.json')` → `app.asar.unpacked/package.json` 경로.
- 하지만 `electron-builder.config.js`에서 `package.json`은 `files` 목록엔 있으나 **`asarUnpack` 목록엔 없음.**
- 즉 `package.json`은 asar **안에**만 있고, unpacked 폴더엔 없음 → `require` 실패 → 빈 문자열.

### 해결 방안 (3가지 옵션)

**옵션 A (권장): main 프로세스에서 버전을 환경변수로 주입**
- `electron/main.cjs`에서 서버 fork 시 `env: { OSOO_APP_VERSION: app.getVersion() }` 추가.
- `diagnosticLogService.getAppVersion()`에서 `process.env.OSOO_APP_VERSION` 우선 확인.
- 가장 깔끔하고 asar 구조 변경 없음.

**옵션 B: package.json을 asarUnpack에 추가**
- `electron-builder.config.js`의 `asarUnpack`에 `'package.json'` 추가.
- 단, asar 구조 변경이라 `AGENTS.md`상 사용자 확인 필요.

**옵션 C: require 경로를 절대 경로로**
- `require(path.join(process.resourcesPath, 'app.asar', 'package.json'))`.
- asar 안의 파일도 읽을 수 있으나 약간 해키함.

**권장**: 옵션 A. main에서 `app.getVersion()`은 정확하고, 환경변수 전달은 표준적.

## 6. 🔴 수정 필요 — 통합 입력 모달 + 유량/재고 cascade 재계산 (사용자 핵심 요구)

### 사용자 요구사항
> **유량**: 과거 날짜의 검침값(raw)을 수정하면 → 그 이후 ~ 오늘까지 계산값(calculated)이 자동 재계산되어야 함.
> **약품/키트**: 사용량/입고량 수정 시 그 이후 재고 자동 재계산. **재고를 직접 수정하면 그날부터 다시 재계산.** (엑셀 수식 체인과 동일)

### 현장 사례 (7/5 로그 증거)
- 7/3 유량 검침값이 비어있는 상태에서 7/4에 값을 넣음 → 7/3 calculated_flow가 `-294720`(마이너스)로 저장됨.
  - 로그: `POST /api/flows/bulk -> 200 | {"results":[{"type":"유입유량계","calculated_flow":-294720},...]}`
- 이후 7/3 raw를 채워 넣어도 **calculated가 갱신되지 않음** ("수정이 안 되고 있어").
- 약품/키트도 과거 수정 시 동일한 cascade 동작이 요구됨.

### 🔴 결함 1: 유량 cascade의 storedFlow 보존이 과잉 (flowRoutes.cjs:87-94)

```js
// flowRoutes.cjs:87-94 (recalculateFlowTypeCascade 내부)
} else if (hasRaw) {
  const storedFlow = row.calculated_flow == null ? null : Number(row.calculated_flow);
  if ((isExplicit || shouldPreserveStoredFlow || row.is_manual || row.is_reset) && storedFlow != null) {
    flow = storedFlow;   // ← 기존 calculated_flow(마이너스 포함)를 그대로 보존!
  } else if (prevRaw != null && Number.isFinite(prevRaw)) {
    flow = Math.round((rawNum - prevRaw) * 10) / 10;
  }
  prevRaw = rawNum;
}
```

**문제**: 사용자가 raw를 수정해도, 이미 calculated_flow가 있으면(그게 -294720이든 뭐든) **그대로 둡니다.** raw가 바뀌었으니 calculated도 재계산되어야 하는데, `isExplicit` 조건 때문에 보존해버림.

**근본 원인**: bulk 라우트(207번 줄)가 **수정한 모든 날짜를 explicitDates에 넣습니다** (`dates.add(date)`). 그런데 explicitDates의 의미가 애매합니다 — "사용자가 직접 calculated를 지정한 날짜"여야 하는데, 지금은 "사용자가 건드린 날짜(=raw만 바꾼 경우도)"까지 explicit으로 취급됩니다. raw만 바꾼 경우엔 calculated를 재계산해야 하는데 보존해버림.

### 🔴 결함 2: 유량 cascade가 마이너스를 허용 (flowRoutes.cjs:91-93)

```js
} else if (prevRaw != null && Number.isFinite(prevRaw)) {
  flow = Math.round((rawNum - prevRaw) * 10) / 10;  // ← raw < prevRaw면 마이너스 허용
}
```

검침값은 누적이라 **오늘 raw < 어제 raw**면 보통 비정상(리셋 안 함). 단건 저장 라우트(222-235번 줄)는 400 에러로 막지만, **bulk 라우트 + cascade는 마이너스를 그냥 저장**합니다.

### ✅ 재고 cascade는 정상 (inventoryCascadeService.cjs + medicine/kitRoutes)

```js
// inventoryCascadeService.cjs:55-63
let runningInventory = Number(previous?.current_inventory || 0);
for (const row of rows) {
  if (normalizedExplicitDates.has(row.date) && row.current_inventory != null) {
    runningInventory = Number(row.current_inventory);  // 재고 직접 수정은 보존
  } else {
    runningInventory = roundInventory(
      runningInventory + Number(row.purchase_amount || 0) - Number(row.usage_amount || 0)
    );  // 입고/사용 수정 시 재계산
  }
  ...
}
```

그리고 medicine/kit bulk 라우트는 **`current_inventory`가 있는 날짜만 explicitDates에 추가** (`medicineRoutes.cjs:88-89`):
```js
if (item.current_inventory !== null && item.current_inventory !== undefined) {
  touched.explicitDates.add(item.date);
}
```

즉 재고 cascade는 "재고 직접 수정 → 보존, 입고/사용 수정 → 재계산"이 올바르게 분리돼 있습니다. **유량도 이 패턴을 따라야 합니다.**

### 해결 방안

**유량 cascade 수정 (flowRoutes.cjs)**:

1. **explicitDates의 의미 재정의**: "사용자가 calculated를 수동 지정한 날짜(is_manual)"만 explicit으로. raw만 바꾼 날짜는 explicit에서 제외.
   - bulk 라우트(207번 줄)에서 `dates.add(date)` 대신, `item.is_manual` 또는 `item.calculated_flow`가 사용자가 명시적으로 넘어온 경우에만 explicit에 추가.
   - 또는 프론트가 "이 날짜는 raw만 바꿨어" vs "calculated까지 직접 정했어"를 구분해서 보내도록.

2. **raw가 바뀐 날짜는 calculated 재계산**: cascade 89번 줄의 조건에서 `isExplicit` 제거 (raw가 있고 prevRaw가 있으면 무조건 `rawNum - prevRaw` 재계산). 단 `is_manual`/`is_reset`/`shouldPreserveStoredFlow`(imported/baseline)는 여전히 보존.

3. **마이너스 방지 (선택)**: bulk/cascade에서 `raw < prevRaw`이고 `is_reset`이 아니면 0 또는 에러 처리. 단, 현장에서 리셋 없이 검침값이 작아지는 경우(고장 등)가 있을 수 있으니 정책 확인 필요.

**검증 시나리오**:
- 7/3 raw 비어있고 7/4 raw=1000 저장 → 7/3 calculated=null(또는 0), 7/4 calculated=1000-이전raw
- 나중에 7/3 raw=900 채워 넣으면 → 7/3 calculated=900-7/2raw, **7/4 calculated=1000-900=100 자동 재계산**
- 7/5 raw=1100 넣으면 → 7/5 calculated=1100-1000=100

**재고는 이미 정상**이지만, 확인 포인트:
- 사용자가 7/3 재고를 직접 수정하면 → 7/3은 보존, **7/4 이후 재고가 7/3 기준으로 재계산**되는지 (cascade의 startDate가 7/3으로 넘어가는지) 검증 필요.

### 통합 입력 모달 측 확인 사항
- `UnifiedRecordModal.jsx`에서 유량 raw 수정 시 프론트가 calculated를 같이 넘기는지, 아니면 raw만 넘기는지. raw만 넘기면 서버가 재계산해야 함.
- 약품/키트 재고 직접 수정 시 `current_inventory` 필드가 bulk payload에 포함되는지.

## 9. 검토 요청사항 (다른 에이전트 대상)

1. **자동업그레이드 정책(섹션 2)**: idle 자동설치를 어느 시점에 둘지 — 다운로드 완료 즉시 vs idle 5분 후 vs 새벽 특정 시간. 그리고 로그인 시 안내 모달의 정확한 문구/타이밍.
2. **single instance lock(섹션 3)**: `electron/main.cjs` 수정은 `AGENTS.md`상 사용자 확인 항목 — 진행 여부.
3. **app_version(섹션 4)**: 옵션 A(환경변수 주입) vs B(asarUnpack 추가) vs C(절대경로) 중 어느 쪽이 기존 구조와 가장 일관성 있는지.
4. **idle 측정 단순화**: 1.0.9의 `before-input-event` 의존을 `powerMonitor.getSystemIdleTime()` 단일 기준으로 바꿀 때, 트레이 최소화 상태에서도 올바르게 측정되는지 검증 필요.
5. **유량 cascade(섹션 6-결함 1)**: "사용자가 raw만 바꾼 날짜"와 "calculated까지 수동 지정한 날짜(is_manual)"를 프론트가 어떻게 구분해서 보낼지 — `UnifiedRecordModal.jsx`의 유량 payload 구조 확인 필요. raw만 바뀌면 서버가 무조건 재계산하는 방향이 맞는지.
6. **유량 마이너스 정책(섹션 6-결함 2)**: bulk/cascade에서 `raw < prevRaw`인 경우 마이너스를 허용할지, 0으로 클램프할지, 아니면 is_reset이 없으면 거부할지 — 현장에서 검침값이 작아지는 정상 케이스(고장/교체)가 있는지 확인 필요.
7. **재고 cascade 검증(섹션 6)**: 사용자가 중간 날짜 재고를 직접 수정했을 때, 그 이후~오늘까지 재고가 자동 재계산되는지 실데이터로 검증 필요 (코드상으로는 정상으로 보이나 확인 권장).
