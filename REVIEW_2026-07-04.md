## 1. 🔴 정정 사항 (중요)

### 정정 1: ESCAPE 버그는 아직 고쳐지지 않았음

계획서에 "즉시 수정 dailyWorkLogHwpxService.cjs의 ESCAPE '\' 버그"가 **todo로만** 있고, 코드엔 **여전히 버그가 남아있음**을 확인했습니다.

```js
// server/services/dailyWorkLogHwpxService.cjs:251-252 (현재 상태, 미수정)
const activeFlowNames = db.prepare(`
  SELECT item_name FROM config_items
  WHERE category = 'flow' AND is_active = 1
    AND item_name NOT LIKE '%\_raw' ESCAPE '\'   // ← 백틱 안 → SQL ESCAPE '' → 에러
    AND item_name NOT LIKE '%\_flow' ESCAPE '\'
  ORDER BY display_order, id
`).all();
```

**7/3 로그에 에러가 0건이었던 이유**: 버그 수정 때문이 **아님**. 7/3 저녁 로그(51줄)는 전부 `GET /api/*/history` 데이터 조회뿐, **HWPX/PDF 내보내기와 매핑 작업을 사용자가 시도조차 안 했기 때문**.

→ ESCAPE 버그 수정은 **지금 바로 해야 하는 1순위**인 상태 유지. (계획엔 맞음, 실행이 안 됐음)

### 정정 2: 유량 매핑 "시작행(369) 날짜 없음" 500 에러가 계획에 빠져있음

7/2 현장 로그에서 확정된 에러 중 **장애 2(유량 매핑 500)** 가 계획서에 명시적으로 없습니다.

- 발생: `POST /api/settings/save-flow-mapping -> 500`, "유량 데이터 시작행(369)의 날짜를 확인할 수 없습니다" (07:37, 07:38 연속)
- "매핑 저장과 임포트 분리" 정책으로 **부분 해결**됨 (설정은 저장, 임포트만 실패). ✅
- 하지만 **시작행 날짜가 없을 때 친절한 안내**(가까운 유효 행 제안)가 없으면 사용자가 "왜 안 되지?" 상태로 남음.

→ 권장: 매핑 임포트 실패 시, 시작행 ±N행 스캔으로 "가까운 날짜 행: 372" 같은 제안을 에러 메시지에 포함.

### 정정 3: 수질 매핑 "다음 단계" 미루는 위험

계획서: "Water key 안정화는 구조 영향이 있어 다음 단계"

**부분 동의**하나, **저장/복원 충돌 정리(레거시 water vs water_mapping)만큼은 1단계에 넣을 것을 권장**합니다.

이유:
- 7/2 로그상 water 매핑은 200 OK로 동작했지만, 이건 **alias 매칭이 우연히 통과한 것**. 한글 장소명/파라미터명에 공백·전각문자 차이가 생기면 조용히 무시됨 (`mappingSettingsService.cjs:349: if (!definition) return false`).
- 복원 로직(`useSettingsViewModel.js:414-424`)이 `water_mapping` + 레거시 `water` 양쪽을 병합하는데, **이건 key를 안 바꿔도 정리할 수 있는 부분**. key 자체를 `paramCode_장소명`으로 바꾸는 건 2단계여도 됨.

→ 권장 분리:
- **1단계**: 레거시 `water` 카테고리 매핑 데이터를 `water_mapping`으로 이관 + 복원 로직 단일화 (key 변경 없이도 가능)
- **2단계**: key를 `paramCode_장소명`으로 안정화

## 2. ✅ 전적으로 동의하는 부분

### 매핑 저장/임포트 분리 정책
> "매핑 설정은 먼저 저장. 엑셀 원본/날짜/범위 문제가 있으면 임포트만 실패 처리. 설정값은 롤백하지 않음."

- 정확히 제 진단의 **문제 5(ensureExcelReady 실패 시 전체 롤백)** 해결책과 일치.
- 사용자 경험 대폭 개선. 강력 지지.

### importProgress 타입별 분리
> "flow / medicine / kit / water 타입별로 분리"

- 제 진단의 **문제 1** 그대로. 7/2 로그에서 4개 매핑을 연속 저장(04:34~04:43)하고 각각 200ms 이내에 import-progress 조회 → 경쟁 위험이 실제 로그로 확인됨.
- 프론트 `useMappingSettings.js:33`의 `getImportProgress()` 시그니처 변경 필요 (type 인자).

### 재고 정책 분리 (admin 신뢰 vs 일반 사용자 제한)
> "admin/설정 임포트: 엑셀 값 그대로 신뢰, 음수 차단 안 함"
> "일반 사용자: 0 미만 경고/제한"

- 현장 업무 현실을 잘 반영한 정책. admin이 과거 데이터를 엑셀로 일괄 임포트할 땐 음수가 정상일 수 있으니 차단하면 안 됨.
- 일반 사용자는 실수 방지. 합리적.

### 자동업데이트 방향 유지
> "실행 시 1회 / 오전 9시 1회 / 수동 확인. 폴링·idle 설치 제거"

- 제 진단의 **문제 A·B·C·D** 해결과 정확히 일치.
- `StatusBar.jsx`의 7개 이벤트 구독 + 능동 설치 버튼도 확인 완료.
- 잔여 제안: `installDownloadedUpdateAndQuit` 호출 전 fork된 서버 자식 프로세스 종료 대기 확인 권장 (문제 B 잔여 위험 최소화).

### 현장 템플릿/배포 확인 방향
> "개발 폴더엔 .hwpx 있음, builder에도 포함됨, AppData 누락 의심"

- 동의. 단, **`validate:asar`로 HWPX 포함 여부를 릴리즈 전에 반드시 확인**해야 함. 7/2 로그의 `REPORT_HWPX_TEMPLATE_MISSING` 404가 설치본에서만 발생했으므로, 빌드 산출물 검증이 선행되어야 원인이 확정됨.
